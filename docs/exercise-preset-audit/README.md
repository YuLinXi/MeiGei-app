## 关键结论

1. 当前 183 条精选库方向正确，已经从 1019 条长尾动作收敛到更适合训练记录工具的规模；不建议恢复旧长尾导入库。
2. 最大的数据问题不是名称，而是协同肌过粗或误配：飞鸟类误带肱三头肌、划船类只标菱形肌、硬拉/架上拉/耸肩误带二头或后三角、腿部复合动作缺少臀/内收/腘绳协同。
3. 自定义/功能性来源的动作需要重点处理：`BB_HIGH_PULL` 建议从肩改归功能性/爆发奥举；`BATTLE_ROPE`、`TURKISH_GET_UP`、`FARMER_CARRY` 的器械或功能性子类不够准确；`BAND_SHOULDER_PASS_THROUGH` 器械应为弹力带。
4. `CABLE_SINGLE_ARM_PULLDOWN` 与 `SINGLE_ARM_LAT_PULLDOWN` 高度重复，建议合并为 alias，避免历史曲线和搜索结果分裂。
5. 非解剖类热身拉伸不高亮肌肉是合理策略；本次仅建议少量命名/器械修正，不建议把猫牛式、下犬式、泡沫轴等强行塞入力量肌群。

## 建议落地顺序

1. 先改明显错误且低争议字段：飞鸟类协同肌、耸肩协同肌、`BAND_SHOULDER_PASS_THROUGH` 器械、`DB_SINGLE_LEG_DEADLIFT` 重复肌群。
2. 再处理复合动作的协同肌补全：划船、硬拉/髋铰链、蹲/腿举/箭步、臀推/臀桥。
3. 最后做会影响用户浏览路径的分类决策：`DIAMOND_PUSH_UP` 是否移到手臂、`BB_HIGH_PULL` 是否移到功能性、`CABLE_SINGLE_ARM_PULLDOWN` 是否并入 alias。
4. 若决定回填 JSON，建议把 `current-preset-exercise-audit.csv` 中 `总评=建议调整` 的建议字段作为 patch 输入，再跑一次 iOS build 或至少用 manifest decode 脚本校验 JSON。

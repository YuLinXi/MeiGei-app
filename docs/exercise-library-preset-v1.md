# 《别练了》预置动作库 V1 评估稿

> 生成日期：2026-06-25  
> 目的：重新定义《别练了》的预置动作库内容，先交由产品评估，再进入 OpenSpec/开发实施。  
> 结论先行：预置动作库不应继续追求 1000+ 全量导入，而应采用「单一精选动作库 + 别名兼容 + 明确移除清单」。

## 1. 背景与判断

当前动作库约 1019 条，包含大量导入动作、版本动作、左右侧动作、热身拉伸、功能性、有氧和课程式动作。这个规模更接近 Keep 这类内容平台，而不是严肃训练记录工具。

《别练了》的目标用户是认真记录训练的人。动作库的第一优先级不是“覆盖所有训练内容”，而是让用户能快速选到主流健身房动作，并让历史曲线、PR、计划回写稳定。

因此建议拆成三层：

```text
预置动作库
  统一展示，覆盖主流力量动作，以及确认保留的热身、拉伸、活动度、功能性动作

别名兼容层
  不作为新动作展示，用于兼容旧 code、旧名称、用户历史、计划引用

移除清单
  不进入新预置动作库；历史中已经出现的记录保留原名或映射显示
```

## 2. 业界参考

主流产品大致分两类：

| 类型 | 代表 | 动作库策略 | 对《别练了》的启发 |
|---|---|---|---|
| 训练记录工具 | Hevy、Strong、StrengthLog、训记 | 以肌群/部位、器械、搜索、自定义动作为主 | 预置库应保持清晰，优先常见力量动作 |
| 动作/课程内容平台 | Keep、JEFIT、MuscleWiki | 动作库大，覆盖课程动作、拉伸、有氧、教学内容 | 可参考覆盖面，但不应照搬为默认动作库 |

## 3. 动作库收录原则

### 3.1 预置动作库收录

预置动作必须满足至少两条：

- 健身房主流动作，用户会反复记录重量、次数、组数。
- 能归入清晰部位和器械。
- 对历史曲线/PR 有长期价值。
- 在训练计划中常被使用。
- 线上现有用户历史已经出现。

预置库优先覆盖：

```text
胸 / 背 / 肩 / 手臂 / 腿 / 臀 / 核心
杠铃 / 哑铃 / 器械 / 史密斯 / 悍马机 / T杠 / 绳索 / 自重 / 壶铃
```

### 3.2 热身、拉伸、活动度、功能性动作收录

确认保留的热身、拉伸、活动度、功能性动作并入同一个预置动作库，不再单独分层。

- 热身、拉伸、泡沫轴、活动度。
- 功能性、爆发、搬运、敏捷。
- 悬挂、弹力带、康复倾向动作。
- 高度长尾但可能被搜索到的动作。

### 3.3 不作为独立预置动作

以下内容不进入新预置动作库：

- 左侧/右侧拆分动作：统一为「单臂」「单腿」或动作本体。
- `版本2`、`版本3`、`进阶版`、`中难度`、`高难度`。
- 只表达负重状态的变体：如“负重俯卧撑”应通过重量字段记录。
- 以计时为主、缺少重量×次数记录价值的有氧训练动作：如跑步、椭圆机、动感单车等，不作为新预置动作；历史中已出现者保留原名展示。
- 课程动作组合：如“XX 接 XX”“XX 训练挑战”。
- 暂无清晰训练记录价值的康复/瑜伽体式。
- HealthKit 泛称：如 `TraditionalStrengthTraining` 不应进入动作库。

## 4. 推荐信息架构

用户看到的是单一预置动作库：

```text
部位：全部 / 胸 / 背 / 肩 / 手臂 / 腿 / 臀 / 核心
器械：全部 / 杠铃 / 哑铃 / 器械 / 绳索 / 史密斯 / 悍马机 / 自重 / 其他
搜索：搜索预置动作库 + 别名兼容层
```

局部细分保留：

```text
胸：上胸 / 中下胸
肩：前束 / 中束 / 后束
```

不建议暴露完整三级树：

```text
胸 -> 胸大肌 -> 上胸
背 -> 菱形肌 / 背阔肌 / 斜方肌 / 竖脊肌
核心 -> 腹直肌 -> 上腹 / 下腹
```

这些可以继续作为内部归位和肌肉高亮数据，但不作为用户导航负担。

### 6.5 实施前零遗漏校验

实施新库前，必须保证：

```text
线上历史动作引用 - 新预置动作库 - 别名兼容表 - 移除清单 = 空集
线上计划动作引用 - 新预置动作库 - 别名兼容表 - 移除清单 = 空集
```

生产库只读查询 SQL：

```sql
SELECT
  CASE WHEN we.builtin_exercise_code IS NULL THEN 'custom' ELSE 'builtin' END AS kind,
  COALESCE(we.builtin_exercise_code, '') AS code,
  we.exercise_name,
  COUNT(DISTINCT we.workout_id) AS workout_count,
  COUNT(ws.id) AS set_count
FROM workout_exercise we
JOIN workout w ON w.id = we.workout_id
LEFT JOIN workout_set ws ON ws.workout_exercise_id = we.id
WHERE w.deleted_at IS NULL
GROUP BY kind, code, we.exercise_name
ORDER BY workout_count DESC, set_count DESC, we.exercise_name;
```

计划引用查询 SQL：

```sql
WITH plan_items AS (
  SELECT item
  FROM workout_plan p
  CROSS JOIN LATERAL jsonb_array_elements(p.items) AS item
  WHERE p.deleted_at IS NULL
)
SELECT
  CASE
    WHEN item ? 'builtinExerciseCode' AND NULLIF(item->>'builtinExerciseCode','') IS NOT NULL THEN 'builtin'
    WHEN item ? 'customExerciseId' AND NULLIF(item->>'customExerciseId','') IS NOT NULL THEN 'custom'
    ELSE 'snapshot'
  END AS kind,
  COALESCE(item->>'builtinExerciseCode', '') AS code,
  COALESCE(item->>'exerciseName', item->>'displayExerciseName', '') AS exercise_name,
  COUNT(*) AS plan_item_count
FROM plan_items
GROUP BY kind, code, exercise_name
ORDER BY plan_item_count DESC, exercise_name;
```

## 7. 推荐实施边界

### 7.1 不需要承担重历史兼容

当前仍处 TestFlight，真实用户极少，历史兼容负担远小于正式上线后。可以做一次较彻底的动作库收敛，但必须保护这两类数据：

- 已有训练历史中的动作引用。
- 已有训练计划中的动作引用。

不建议再把“所有 TestFlight 出现过的 code 永久冻结”作为硬规则。更合理的是：

```text
正式上架前：
  可以重命名/合并/隐藏/删除低风险导入动作
  但必须用线上历史覆盖校验保护已有用户数据

正式上架后：
  code 一经发布给真实用户，即冻结
```

### 7.2 推荐下一步

1. 你先评估本文件中的动作范围，重点看：
   - 预置动作库是否过少或过多。
   - 哪些保留/移除结论需要调整。
   - 胸/背/肩/手臂/腿/臀/核心的标准命名是否符合你的习惯。
2. 评估通过后，再更新 OpenSpec change `simplify-exercise-library-ia`：
   - 增加“预置动作库 V1 清单”作为 design 附件或数据任务。
   - 明确别名兼容表和线上覆盖校验为实施前门禁。
3. 最后再进入开发：
   - 生成预置动作、别名兼容、移除清单 manifest。
   - 重写或清理当前 imported 动作。
   - 调整 iOS 动作库展示。
   - 跑线上历史覆盖脚本，确保遗漏为 0。

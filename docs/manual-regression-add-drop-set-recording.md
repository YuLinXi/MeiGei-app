# add-drop-set-recording 完整验收清单

本文档是 OpenSpec change `add-drop-set-recording` 的验收入口，覆盖训练录入、统计、计划处方、同步、Team、兼容与无障碍。用例同时面向人工回归与 XcodeBuildMCP 模拟器自动化：自动化执行时按 `id` 顺序跑，优先处理 `类型=sim-ui` 与 `类型=unit`。

## 自动化约定

- iOS 工程：`ios/DontLift/DontLift.xcodeproj`
- scheme：`DontLift`
- 推荐模拟器：`iPhone 17 Pro`
- 推荐构建参数：`CODE_SIGNING_ALLOWED=NO`
- 执行器策略：优先用 accessibility label / 可见文本定位元素；只在 UI 树缺少稳定元素时使用坐标。
- 文案约束：用户可见 UI 使用「组」描述递减组内部项目；不得出现旧文案「分段」「添加一段」「第 N 段重量」「第 N 段次数」。
- 数据断言策略：UI 可见信息用模拟器截图或 UI snapshot 断言；`setType=drop`、`segments`、`setPrescriptions`、脱敏字段等结构化断言优先用现有单元测试或调试数据探针验证。
- 不可由单台模拟器稳定验证的项标记为 `backend`、`multi-device` 或 `manual-device`，不应阻断 `sim-ui` 自动化主流程。

## 全局前置条件

- App 可在 DEBUG 模拟器启动并完成登录；本地训练和计划用例可在离线 SwiftData 数据下执行。
- 需要 Team、同步或后端脱敏验证时，后端以开发配置启动，并使用 dev token 登录测试账号。
- 每轮自动化前建议清理 App 数据，避免历史训练影响自适应预填、PR 和计划回写判断。
- 自动化脚本创建测试动作时优先选择内置动作；若当前动作库数据变化，只要求选择任意可记录重量和次数的动作。

## AC-01 底部按钮与快速新增递减组

- 类型：`sim-ui`
- 前置：新建或打开一条进行中的训练，至少添加 1 个动作。
- 步骤：
  1. 在动作卡底部找到「递减组」和「加一组」两个入口。
  2. 点击「加一组」，输入 `70kg x 8`。
  3. 点击动作卡底部「递减组」。
- 断言：
  - 「递减组」按钮在「加一组」左侧，尺寸更紧凑、颜色更弱。
  - 「加一组」按钮占据剩余宽度，边框和文字使用次主题色。
  - 新增 1 个父级正式组，父级文案显示「递减组」。
  - 递减组显示内部组数统计，例如 `2组`。
  - 第 1 个内部组继承上一正式组的 `70kg x 8`；第 2 个内部组为空。
  - 点击「递减组」后不自动选中重量输入框，不弹出数字键盘。
  - UI 中不出现「分段」「添加一段」等旧文案。
- XcodeBuildMCP 备注：
  - 通过 UI snapshot 查找文本「递减组」「加一组」「2组」。
  - 点击后立即再次截屏，确认没有自定义数字键盘面板出现。

## AC-02 递减组内部添加、删除、折叠与文案

- 类型：`sim-ui`
- 前置：存在一个展开态递减组。
- 步骤：
  1. 点击递减组内部「添加」。
  2. 删除其中一个内部组。
  3. 点击递减组标题区域，切换折叠；再次点击恢复展开。
- 断言：
  - 「添加」按钮与内部重量框左侧对齐，按钮文案为「添加」。
  - 点击「添加」后新增内部组，但不自动选中重量输入框，不弹出键盘。
  - 删除后内部组编号重新从 1 顺序展示，父级组编号不变化。
  - 折叠态保留「递减组」和 `N组` 统计，不使用常规 chevron 图标表达展开状态。
  - 展开态左侧区分线使用弱化主色；内部组保持缩进，与普通组形成区分。
  - 内部组输入框尺寸小于普通组输入框，整体上下间距紧凑。
- XcodeBuildMCP 备注：
  - 自动化只断言文本、元素存在和键盘未出现；左侧线颜色、输入框尺寸建议用截图人工复核或视觉快照比较。

## AC-03 任意重量方向录入

- 类型：`sim-ui`、`unit`
- 前置：存在一个递减组，至少 3 个内部组。
- 步骤：
  1. 录入 `50kg x 10`、`60kg x 8`、`55kg x 6`。
  2. 保存或保持当前训练页。
- 断言：
  - App 不提示重量方向错误。
  - 递减组名称仍显示「递减组」。
  - 不出现「递增组」「混合组」等派生名称。
  - 模型保存有序 `segments`，不额外持久化方向判断结果。
- XcodeBuildMCP 备注：
  - UI 自动化可验证无错误提示和文案不变。
  - 结构化保存结果建议由单元测试覆盖。

## AC-04 普通组与递减组互转

- 类型：`sim-ui`
- 前置：动作卡中存在普通正式组 `80kg x 5`。
- 步骤：
  1. 打开该普通组右侧 `...` 菜单。
  2. 选择「改为递减组」。
  3. 再打开该递减组右侧 `...` 菜单，选择「改回普通组」并确认。
- 断言：
  - 普通组菜单提供「改为递减组」。
  - 转换后父级显示「递减组」，第 1 个内部组为 `80kg x 5`，并追加 1 个空白内部组。
  - 改回普通组前出现确认弹窗。
  - 确认后使用第一个有效内部组回填父级普通组重量和次数，其它内部组被移除。
- XcodeBuildMCP 备注：
  - 菜单项用文本定位。
  - 弹窗断言包含“保留第一组有效数据”语义即可，不依赖完整长句。

## AC-05 热身组互斥

- 类型：`sim-ui`
- 前置：动作卡中存在一个普通正式组。
- 步骤：
  1. 将普通组切换为热身组。
  2. 打开热身组右侧 `...` 菜单。
  3. 将热身组切回正式组，再打开菜单。
- 断言：
  - 热身组菜单不直接展示「改为递减组」。
  - 切回正式组后才可选择「改为递减组」。
  - 递减组本身不提供标为热身组的入口。

## AC-06 自定义数字键盘焦点序列

- 类型：`sim-ui`
- 前置：动作卡中存在一个递减组，内部至少 2 组。
- 步骤：
  1. 点击第 1 个内部组重量。
  2. 通过键盘「下一项」依次移动焦点。
  3. 在任一内部组焦点状态点击键盘「加一组」。
- 断言：
  - 焦点顺序为：第 1 组重量 -> 第 1 组次数 -> 第 2 组重量 -> 第 2 组次数。
  - 在递减组内部焦点时点击键盘「加一组」，新增的是父级普通正式组，不是递减组内部组。
  - 新增普通组后焦点落在该普通组重量，保持高频录入体验。
- XcodeBuildMCP 备注：
  - 通过输入框高亮、键盘可见和新增行文本综合断言。

## AC-07 完成计数、容量、次数、清理规则

- 类型：`unit`、`sim-ui`
- 前置：新建训练并添加动作。
- 步骤：
  1. 完成一个递减组：`80kg x 8`、`60kg x 6`、`45kg x 8`。
  2. 再创建一个未完成递减组。
  3. 创建一个已完成递减组，但保留一个完全空白内部组。
  4. 结束训练。
- 断言：
  - 第 1 个递减组让已完成组数增加 1。
  - 训练量增加 `80*8 + 60*6 + 45*8`。
  - 总次数增加 `8 + 6 + 8`。
  - 未完成递减组整体不进入完成记录，不影响训练量、PR、历史曲线或计划回写。
  - 完全空白内部组被清理或不参与统计。
  - 逻辑组数按父级递减组计 1，不按内部组数量计数。
- XcodeBuildMCP 备注：
  - UI 可断言完成页/详情页展示；精确统计优先由单元测试断言。

## AC-08 PR、历史曲线、详情与海报

- 类型：`unit`、`sim-ui`
- 前置：存在低于 `90kg` 的历史 PR。
- 步骤：
  1. 完成递减组：`90kg x 5`、`70kg x 8`。
  2. 结束训练，查看 PR、训练详情、历史曲线和分享海报。
- 断言：
  - PR 使用递减组内部最大重量 `90kg`。
  - PR 庆祝、动作 PR 副标、训练详情 PR 标记一致。
  - 训练详情显示父级「递减组」和每个有效内部组。
  - 分享海报容量按所有有效内部组计算。
  - 海报动作高光行使用最大重量内部组作为顶组，并表达该递减组包含更多内部组。

## AC-09 Workout 同步与旧数据兼容

- 类型：`unit`、`backend`、`multi-device`
- 前置：后端开发环境可用，至少两个测试账号或两个测试设备。
- 步骤：
  1. 设备 A 创建并同步包含递减组的训练。
  2. 设备 B 拉取该训练。
  3. 构造旧 workout payload：缺少 `segments` 字段。
  4. 模拟两端同时编辑同一 workout。
- 断言：
  - 后端 `workout_set.segments` 持久化为 jsonb，默认空数组。
  - push/pull 原样收发 `setType=drop` 与 `segments`。
  - 设备 B 可看到同一父级递减组和所有内部组。
  - 旧数据缺少 `segments` 时按普通组或空 `segments` 兼容展示，父级重量/次数仍参与普通统计。
  - 冲突仍按 Workout 聚合根 last-write-wins，不做逐内部组 merge。

## AC-10 保存训练为计划模板

- 类型：`sim-ui`、`unit`
- 前置：完成一条包含递减组的无计划训练。
- 步骤：
  1. 在完成后的训练详情选择「保存为计划模板」。
  2. 打开新计划详情。
  3. 从该计划开始训练。
- 断言：
  - 新计划对应动作写入 `setPrescriptions`。
  - 递减组处方包含已完成递减组的有效内部组。
  - 计划列表或详情摘要展示组数、次数、顶组重量，并提示含递减组。
  - 从该计划开始训练时生成递减组父级 set，内部组按处方落值，父级 `completed=false`。

## AC-11 严格计划预填递减组

- 类型：`sim-ui`、`unit`
- 前置：存在严格模式计划，动作处方包含递减组 `80kg x 8`、`60kg x 6`。
- 步骤：
  1. 打开计划详情。
  2. 点击「开始训练」。
- 断言：
  - 训练页生成一个递减组父级 set。
  - 内部组分别落值为 `80kg x 8` 与 `60kg x 6`。
  - 新建组均为未完成状态，需要用户主动勾选。
  - 严格模式完成后不回写来源计划，完成页不展示回写回执。

## AC-12 自适应计划历史优先与回写

- 类型：`sim-ui`、`unit`
- 前置：存在自适应计划，且同动作历史中有已完成递减组 `85kg x 6`、`65kg x 8`。
- 步骤：
  1. 从自适应计划开始训练。
  2. 完成一个新的递减组 `90kg x 5`、`70kg x 8`。
  3. 结束训练，查看完成页回写回执。
  4. 点击「撤销此次更新」。
- 断言：
  - 开始训练优先使用上次 completed 实绩递减组结构。
  - 无历史时才回退计划 `setPrescriptions`；再无处方时回退 `suggested*`。
  - 回写只依据本次 completed 正式组。
  - `suggestedSets` 按父级逻辑组只增不减。
  - `suggestedWeightKg/suggestedReps` 使用最大重量内部组 `90kg x 5`。
  - `setPrescriptions` 保留完整递减组结构。
  - 撤销后来源计划恢复到训练前状态，并重新标脏等待同步。

## AC-13 训练模板新建手动递减组处方

- 类型：`sim-ui`、`unit`
- 前置：计划列表可进入，新建计划数据为空。
- 步骤：
  1. 在计划列表点击添加入口，新建训练模板。
  2. 点击「添加动作」，选择任意动作。
  3. 打开该动作的处方编辑。
  4. 验证默认普通处方为 4 组、10 次、重量为空。
  5. 点击「递减组」添加递减组处方，录入 `80kg x 8`、`60kg x 6`。
  6. 点击递减组内部「添加」，再删除一个内部组。
  7. 保存计划。
  8. 从该计划开始训练。
- 断言：
  - 处方编辑区有「组处方」和组数统计。
  - 可添加普通组和递减组。
  - 递减组处方显示「递减组」和 `N组`。
  - 递减组处方内只展示重量、次数、添加和删除；不展示完成勾选、休息、备注或训练中完成状态。
  - 保存后的计划项 `setPrescriptions` 包含 `setType=drop`。
  - `setPrescriptions` 内部组顺序、重量、次数与用户录入一致。
  - `suggestedSets/suggestedReps/suggestedWeightKg` 同步更新，用于列表摘要和旧端兼容。
  - 下次从计划开始训练时生成对应递减组父级 set。
- XcodeBuildMCP 备注：
  - 自动化可用文本「组处方」「加一组」「递减组」「添加」「保存」驱动。
  - 若动作选择器内容变化，只需选择任意第一个可用动作。

## AC-14 计划详情编辑递减组处方

- 类型：`sim-ui`、`unit`
- 前置：已有计划，至少一个动作存在普通处方。
- 步骤：
  1. 打开计划详情。
  2. 编辑某动作处方。
  3. 将其中一组改为递减组。
  4. 调整递减组内部重量/次数，新增并删除内部组。
  5. 保存并返回计划详情。
- 断言：
  - 普通组可转换为递减组，递减组也可转换回普通组。
  - 删除处方后剩余处方重新顺序展示。
  - 保存后 `setPrescriptions` 更新。
  - 计划详情“下次有效处方”预览展示递减组结构。
  - 计划列表摘要仍使用兼容的 `suggested*` 字段展示简要强度。

## AC-15 旧计划回退与新增动作默认 4x10

- 类型：`unit`、`sim-ui`
- 前置：存在一个没有 `setPrescriptions` 的旧计划项。
- 步骤：
  1. 从旧计划开始训练。
  2. 在计划详情新增一个动作。
  3. 从更新后的计划再次开始训练。
- 断言：
  - 旧计划项继续按 `suggestedSets/suggestedReps/suggestedWeightKg` 生成普通组。
  - 计划详情新增动作默认保存 `suggestedSets=4` 与 `suggestedReps=10`。
  - 下次训练生成 4 个 `reps=10`、`completed=false` 的普通组。
  - 用户仍可在新增动作处方编辑区把任意普通组改为递减组或追加递减组处方。

## AC-16 计划复制、Team 分享与 Fork 脱敏

- 类型：`unit`、`backend`、`sim-ui`
- 前置：存在含递减组处方的计划，且递减组内部有重量。
- 步骤：
  1. 复制该计划为个人新计划。
  2. 分享该计划到 Team。
  3. 用 Team 成员账号 Fork 该 Team 计划。
- 断言：
  - 复制/Fork 后保留动作、组数、次数和 `setPrescriptions` 结构。
  - 新计划模式默认 `adaptive`。
  - 所有重量字段清空：顶层 `suggestedWeightKg`、普通处方重量、递减组内部重量、旧 payload 中的 `weightKg/weight`。
  - 内部组次数保留。
  - Fork 后新计划默认进入接收者「未分组」。
  - 发布者的分组结构不复制给接收者。

## AC-17 Team 打卡展示

- 类型：`unit`、`backend`、`sim-ui`
- 前置：用户属于至少一个 Team，存在包含递减组的已完成训练。
- 步骤：
  1. 自动或手动分享训练打卡到 Team。
  2. 打开 Team 打卡列表和打卡详情。
- 断言：
  - checkin summary 中保留递减组 set type 与内部组列表。
  - Team 摘要总组数按父级递减组计 1。
  - 容量按所有有效内部组展开计算。
  - 打卡详情能看到「递减组」和每个有效内部组。
  - 旧摘要缺少 set type 或 `segments` 时，客户端按普通组兼容展示。

## AC-18 VoiceOver 与无障碍标签

- 类型：`sim-ui`、`manual-device`
- 前置：存在包含递减组的训练录入页、训练详情和 Team 打卡详情。
- 步骤：
  1. 在训练录入页聚焦递减组父级行。
  2. 逐个聚焦递减组内部重量和次数输入框。
  3. 聚焦「添加」、删除、折叠入口。
  4. 打开训练详情或 Team 打卡详情。
- 断言：
  - 父级可朗读「递减组」和第几组。
  - 内部输入框可区分「第 N 组重量」与「第 N 组次数」。
  - 「添加」入口可朗读为给当前递减组添加内部组。
  - 删除入口可朗读被删除的是第几组。
  - 详情页和 Team 打卡详情中的递减组明细可被完整朗读。
  - 无障碍文案不使用「第 N 段重量」「第 N 段次数」。
- XcodeBuildMCP 备注：
  - 模拟器 UI snapshot 可检查 accessibility label；真实 VoiceOver 发音需人工真机确认。

## AC-19 视觉与触感反馈

- 类型：`sim-ui`、`manual-device`
- 前置：存在动作卡底部按钮和递减组内部「添加」按钮。
- 步骤：
  1. 连续点击底部「递减组」和「加一组」。
  2. 连续点击递减组内部「添加」。
  3. 真机上重复步骤 1 和 2。
- 断言：
  - 按钮点击时不出现从上往下移动的按压动画。
  - 高频点击不容易误触相邻按钮；「递减组」与「加一组」触发区域清晰分离。
  - 「递减组」按钮颜色弱化；「加一组」按钮颜色略强，边框和文字同为次主题色。
  - 递减组内部「添加」按钮紧凑、弱化，且与重量输入框左侧对齐。
  - 真机点击「递减组」「加一组」「添加」有轻量触感反馈。
- XcodeBuildMCP 备注：
  - 模拟器只能验证按钮 frame 无异常位移和 UI 变化；触感反馈必须真机或代码审查确认。

## 建议自动化执行顺序

1. 运行单元测试：覆盖模型、统计、计划预填、计划回写、Team 摘要、脱敏和兼容。
2. 启动模拟器 App，执行 `AC-01` 到 `AC-06` 的训练录入主链路。
3. 在同一模拟器继续执行 `AC-10` 到 `AC-15` 的计划闭环。
4. 后端可用时执行 `AC-09`、`AC-16`、`AC-17`。
5. 最后执行 `AC-18`、`AC-19` 的 UI snapshot 与真机补充验证。

## 通过标准

- P0 用例全部通过。
- P1 用例若因环境不可用跳过，必须记录原因和替代验证方式。
- 不得出现旧文案「分段」「添加一段」「第 N 段重量」「第 N 段次数」。
- 递减组结构在训练、计划、同步、Team、详情和海报之间保持一致。
- 任何统计数字必须能从原始 `WorkoutSet` 与 `segments` 重算，不能依赖冗余持久化统计。

## Requirements
### Requirement: 动作浏览分类轴（两级肌群）

本 Requirement 由「两级肌群（父级 → 高亮叶级）」升级为单一解剖三级树后，进一步明确其角色：`L1 部位 → L2 肌肉 → L3 肌头/肌区` SHALL 作为动作库内部分类、高亮归位和统计归并的真源；用户默认浏览体验 MUST NOT 强制暴露完整三级树，也 MUST NOT 把全量导入动作同等放入默认动作选择列表。

- **L1 部位** MUST 收敛为 11 个：解剖类 `胸/背/肩/手臂/腿/臀/核心/颈部`（8）+ 非解剖类 `有氧/功能性/热身拉伸`（3）。原以单块肌肉冒充父级者 MUST 降级为 L2：`二头/三头/前臂 → 手臂`、`小腿 → 腿`、`斜方肌 → 背`；`颈部` 保持独立 L1。用户默认浏览入口不要求展示低频 L1，但内部归位枚举 MUST 保持完整。
- **L2 肌肉** SHALL 为每个解剖 L1 声明其肌肉列表（如 胸→胸大肌；背→背阔肌/斜方肌/菱形肌/竖脊肌；手臂→肱二头肌/肱三头肌/前臂；腿→股四头肌/腘绳肌/内收肌/小腿；臀→臀大肌/臀中肌；核心→腹直肌/腹斜肌）。非解剖 L1 无 L2。
- **L3 肌头/肌区** MUST 按需配置（不强配）：仅给用户真会区分的肌肉配（胸大肌→上胸/中下胸；三角肌→前束/中束/后束；腹直肌→上腹/下腹），其余 L2 的 L3 留空。
- 力量类 L1/L2/L3 MUST 与 `MuscleRegion`（高亮叶级）对齐为同一棵树（见「浏览与高亮同源」）；非解剖 L1 MUST 映射为空 region、不显高亮。存储层 `historyKey`（`builtinCode ?? customId ?? name`）计算式 MUST NOT 因本 change 改变。
- 用户默认浏览 MUST 以单一预置动作库为对象，筛选主轴 SHALL 为 `部位 + 器械 + 搜索`。完整 L2/L3 树 SHALL 作为局部细分或高级筛选能力存在，而不是所有动作的默认导航负担。
- UI MUST NOT 使用“核心库/扩展库/扩展动作”作为用户可见分层；确认保留的热身、拉伸、活动度、功能性动作 SHALL 并入同一预置动作库。

#### Scenario: L1 内部枚举收敛为 11 个真部位
- **WHEN** 枚举内部浏览归位 L1
- **THEN** 恰为 11 个（解剖 8 + 非解剖 3）；不存在「二头/三头/前臂/小腿/斜方肌」作为 L1（它们为某 L1 的 L2）

#### Scenario: 三级逐级归属
- **WHEN** 取「胸 → 胸大肌 → 上胸」一条链
- **THEN** L2「胸大肌」属 L1「胸」、L3「上胸」属 L2「胸大肌」；任一级归属非法则校验失败

#### Scenario: 默认浏览不暴露完整三级树
- **WHEN** 用户进入动作选择页且未搜索、未打开高级筛选
- **THEN** 系统默认展示单一预置动作库，并以部位和器械为主筛选；完整 L2/L3 树不作为默认导航强制展开

#### Scenario: 局部保留用户有感知的细分
- **WHEN** 用户在动作库浏览「胸」或「肩」
- **THEN** UI 可直接展示「胸→上胸/中下胸」「肩→前束/中束/后束」；内部归位、高亮、校验仍保留完整 L2 节点（胸大肌、三角肌）

### Requirement: 部位子分类（三层浏览结构）

`BuiltinExercise.subcategory` 由「父级内子分类」**重锚为 L3 肌头/肌区**：非空时其值 MUST 属于该动作所归 L2 肌肉的合法 L3 列表；为空表示未细分到肌头（归该 L2「全部」）。L3 MUST 仅在其 L2 作用域内有效，MUST NOT 跨 L2 复用。

非解剖 L1「功能性」无肌肉/高亮，但 SHALL 在其枝下提供 5 个浏览子类用于下钻：`爆发奥举 / 抗旋稳定 / 髋臀激活 / 负重搬运 / 动态控制`（命名 MUST 避开「核心」的「核心稳定」）。非解剖 L1「有氧」移除「间歇·Tabata」后仅余稳态有氧，其子类列表 MUST 为空、不出现二级。功能性动作逐条归子类属数据债、MAY 分批；未归类者落「全部」、MUST NOT 阻塞浏览。

#### Scenario: 胸大肌下钻到肌头
- **WHEN** 用户在左栏选「胸 → 胸大肌」
- **THEN** 其下展开「上胸 / 中下胸」可三级收窄；上斜卧推类 `subcategory`=「上胸」，平/下斜类=「中下胸」

#### Scenario: 功能性可下钻
- **WHEN** 用户在左栏选「功能性」
- **THEN** 其下展开「爆发奥举/抗旋稳定/髋臀激活/负重搬运/动态控制」；已归类动作进对应子类，未归类仍可在「全部」浏览到

#### Scenario: 有氧不再细分
- **WHEN** 读取「有氧」子类列表
- **THEN** 为空；不出现二级，且不含「间歇·Tabata」

### Requirement: 器械类型扩展与拆细

系统 SHALL 把 `EquipmentType` 从 6 类扩展到约 11 类，新增 `史密斯`、`悍马机`、`T杠`、`弹力带`、`悬挂`（TRX/吊环）、`其他`。悍马机、史密斯、T杠 MUST 作为独立类型，从原「器械/杠铃」拆出。所有 rawValue MUST 为中文且无重复。现有内置动作 MUST 按新粒度重新标注 `equipmentType`，且重标注 MUST NOT 改变其 `code`。

#### Scenario: 史密斯动作重标注但 code 不变
- **WHEN** 现有 `SMITH_BENCH_PRESS` 的 `equipmentType` 由「器械」改为「史密斯」
- **THEN** 其 `code` 保持 `SMITH_BENCH_PRESS` 不变，依赖 code 的历史归并不受影响

#### Scenario: 器械轴可独立筛选
- **WHEN** 用户在器械轴选择「弹力带」
- **THEN** 列表过滤为 `equipmentType == "弹力带"` 的动作，与部位轴正交叠加

### Requirement: 内置动作 code 生成与冻结

每个内置动作 SHALL 拥有唯一、稳定、随包发布的 `code`，作为 `historyKey` 第一主键。新导入动作的 `code` MUST 按「`[器械前缀_]动作主体[_修饰]` 语义英文蛇形」规则由中文名生成，与训记内部序号无关；真碰撞时 MUST 以确定性规则追加 `_2`/`_3`。系统 MUST 维护一份 `中文名 → code` 映射作为唯一权威源。已发布的 `code` MUST NOT 被重算或变更。

#### Scenario: code 全库唯一
- **WHEN** 校验内置动作集（现有 153 + 新导入 928）
- **THEN** 所有 `code` 唯一，无重复

#### Scenario: code 与训记序号无关
- **WHEN** 训记动作表的序号发生变化或某动作下架
- **THEN** 别练了已发布动作的 `code` 不受影响、保持不变

### Requirement: 训记动作全量导入与去重

系统 MUST NOT 再把训记或其他第三方标准动作表当作必须全量导入的用户可见动作库。内置动作库 SHALL 收敛为《别练了》预置动作库 V1，优先覆盖主流健身房力量动作、已确认保留的热身/拉伸/活动度/功能性动作，以及线上真实历史已经出现且需要继续解析的动作。

- 导入来源 MAY 继续作为候选数据源，但进入预置动作库前 MUST 经过标准名、器械、分类、肌肉区和历史覆盖校验。
- 命中现有标准动作或合并规则的导入动作 MUST 复用标准动作或进入别名兼容表，MUST NOT 新建重复条目。
- 计时型有氧训练动作（如跑步、椭圆机、划船机、动感单车、跳绳）MUST NOT 进入新预置动作库；历史中已出现者 MUST 按历史保留策略展示，不得丢失记录。
- `TraditionalStrengthTraining` 等 HealthKit 泛称 MUST NOT 进入预置动作库。
- 清单、合并、移除和线上兼容的评估依据 SHALL 记录在 `docs/exercise-library-preset-v1.md`，实现时 SHALL 生成结构化 manifest 供 iOS 运行时加载。

#### Scenario: 不再全量展示导入动作
- **WHEN** 构建新的内置动作 manifest
- **THEN** 系统只包含 V1 预置动作库确认保留的动作、必要别名兼容和历史保留策略，不把 1000+ 导入动作全量作为用户可选动作展示

#### Scenario: 概念重叠进入别名兼容
- **WHEN** 导入或历史中存在「绳索十字夹胸」「器械飞鸟」「哑铃轮换弯举」等概念重叠动作
- **THEN** 系统 MUST 分别映射到「绳索夹胸」「蝴蝶机夹胸」「哑铃交替弯举」，不得生成重复标准动作

#### Scenario: 计时型有氧历史不丢失
- **WHEN** 历史记录中存在跑步、椭圆机、划船机、动感单车或跳绳
- **THEN** 系统 MUST 能展示已有历史记录；但这些动作不出现在新建训练的预置动作选择列表中

### Requirement: 同名动作历史自动合并

导入内置动作时，若某内置动作的 `name` 撞上用户已有的手填/自定义动作名（其历史此前按 `name` 归并），系统 SHALL 执行一次性本地迁移，把这些旧记录改挂到该内置动作的 `code`，使同一动作的历史曲线连续。该迁移 MUST 仅在本地进行，MUST NOT 新增同步实体或改动 LWW/幂等约定，且 MUST 幂等（可安全重跑）。

#### Scenario: 旧手填记录并入新内置动作
- **WHEN** 用户过去以手填名「战绳」记录过训练，现导入内置「战绳」(code `CARDIO_BATTLE_ROPE`)
- **THEN** 旧记录的 `historyKey` 迁移为指向 `CARDIO_BATTLE_ROPE`，历史曲线连续不断裂

#### Scenario: 无同名不动数据
- **WHEN** 用户无与任何新内置动作同名的历史/自定义动作
- **THEN** 迁移不改动任何记录

#### Scenario: 迁移幂等
- **WHEN** 同名合并迁移被重复执行
- **THEN** 结果与执行一次相同，无重复迁移或数据损坏

### Requirement: 浏览与高亮同源（单一解剖树 + 优雅回退）

系统 SHALL 以**同一棵解剖树**驱动浏览归位与肌群高亮，二者 MUST 同源于动作的 `primaryRegions`/`secondaryRegions`（+ `subcategory` 肌头），MUST NOT 维护两套脱节的分类。

- **浏览归位**：L1 取 `category`（显式）；L2 由 `primaryRegions` 派生（命中哪块肌肉）；L3 取 `subcategory`（显式、按需）。
- **高亮染色**：把目标区染到**美术支持的最深一级**；美术不支持该级时 MUST **优雅回退**到最近可染父级（如美术未拆上/下胸 → 染整块胸；三头未拆头 → 染整三头），MUST NOT 报错或留白崩溃。
- `MuscleRegion` MUST 补齐与树对齐所需的缺失区（至少 `deltSide` 肩中束、`rhomboids` 中背/菱形）。新增区的美术补绘可分批（归 `muscle-map-detailed-art`），补绘前按优雅回退降级。
- `primaryRegions` 为空的动作（如未回填的导入条、非力量动作）MUST 落「L1 → 全部」可浏览、不显高亮，回填后自动归到 L2/L3。

#### Scenario: 高亮回退到可染父级
- **WHEN** 某动作目标为「上胸」但美术尚未拆分上/下胸
- **THEN** 高亮染整块胸（回退父级），不报错；待美术补 L3 区后自动染到「上胸」

#### Scenario: 浏览与高亮不脱节
- **WHEN** 某动作 `primaryRegions` 命中「胸大肌」
- **THEN** 它在浏览中归「胸 → 胸大肌」段，且高亮染胸大肌区——二者同源、不出现「浏览在胸、高亮在别处」

#### Scenario: 缺区补齐
- **WHEN** 校验 `MuscleRegion` 全集对树的覆盖
- **THEN** 含 `deltSide`（肩中束）、`rhomboids`（中背）等原缺失区，使每个 L2/L3 力量节点有可归属的高亮区（美术可后补）

### Requirement: 发布前内置动作去噪清理

系统 SHALL 在正式上架前，对内置动作库做去噪清理与收敛。因当前仍处 TestFlight 阶段，未正式发布或无线上引用风险的导入动作 MAY 直接移除、改名或合并；但任何线上训练历史或计划引用到的动作 MUST 被新预置动作库、别名兼容表或历史保留策略覆盖。正式上架后，已发布给真实用户的 `code` MUST NOT 被删除、复用或改为其他语义。

清理 SHALL 覆盖以下类别：

1. **版本号动作**：`版本2/版本3/进阶版/中难度/高难度` 等仅表达版本或难度的动作 MUST 删除或并入本体。
2. **左右侧重复动作**：左侧/右侧拆分动作 MUST 统一为「单臂」「单腿」或动作本体。
3. **负重状态变体**：如“负重俯卧撑”这类只表达负重状态的变体 MUST 通过重量字段记录，不作为独立标准动作。
4. **课程式组合动作**：如“XX 接 XX”“XX 训练挑战” MUST 不进入预置动作库。
5. **纯计时动作**：跑步、椭圆机、划船机、动感单车、跳绳等 MUST 不进入新预置动作库。
6. **已明确移除动作**：`杠铃单臂推`、`派克俯卧撑`、`扎特曼弯举`、`JM 推`、`西西深蹲`、`哑铃腿弯举`、`腿举机提踵`、`弹力带蚌式`、`平板臂屈伸`、`四足向后旋转`、`TraditionalStrengthTraining`、`沙发伸展` MUST 不进入预置动作库。

#### Scenario: 版本动作不进入新库
- **WHEN** 库中存在同一动作本体及其 `版本2/版本3` 条目
- **THEN** 新 manifest MUST 只保留本体或明确标准动作，版本条目不得作为独立预置动作展示

#### Scenario: 已确认移除动作不可新建
- **WHEN** 用户打开动作选择页并搜索「扎特曼弯举」「JM 推」「西西深蹲」或 `TraditionalStrengthTraining`
- **THEN** 系统不得把这些条目作为预置动作返回；历史已有记录按历史快照或兼容策略展示

#### Scenario: 上线前校验保护已有引用
- **WHEN** 某待删除或待合并动作存在生产训练历史或计划引用
- **THEN** 实施前 MUST 为其补充标准动作、别名映射或历史保留策略；否则不得删除或上线新 manifest

### Requirement: 预置动作库 V1 清单

系统 SHALL 以 `docs/exercise-library-preset-v1.md` 作为预置动作库 V1 的产品评估源，并以生成后的结构化 manifest 作为 iOS 运行时实现源。manifest MUST 至少表达标准名、`code`、器械、部位、细分、主动肌区、协同肌区、别名、legacy code 和历史保留状态。

V1 清单 MUST 遵守以下已确认结论：

- 胸部保留 `下斜史密斯卧推`、`悍马机卧推`、`悍马机推胸`；`悍马机卧推` 与坐姿 `悍马机推胸` MUST 独立。
- `绳索十字夹胸` MUST 合并到 `绳索夹胸`；`器械飞鸟/蝴蝶机飞鸟` MUST 合并到 `蝴蝶机夹胸`。
- `高位下拉` 标准名 MUST 保留，默认语义为宽握；`宽握高位下拉` MUST 作为别名合并到 `高位下拉`。
- 肩部 `面拉` 标准名 MUST 统一为 `绳索面拉`，主肩后束，归肩。
- 只保留 `蝴蝶机反向飞鸟` 标准名，不保留 `反向蝴蝶机` 标准动作。
- `直杆绳索弯举` 与 `绳索弯举` MUST 作为两个独立动作；`坐姿绳索拉杆二头弯举` MUST 独立保留。
- `器械倒蹬机` MUST 作为独立器械动作，不并入 `腿举`。
- `罗马尼亚硬拉` MUST 主分类归腿、副臀；`单腿哑铃硬拉` MUST 主臀副腿；`髋外展` MUST 主臀副腿。
- `农夫行走` 与 `壶铃摆荡` MUST 归功能性。
- `坐姿腿屈伸` MUST 保留为腿部主流动作。

#### Scenario: 清单生成包含关键保留动作
- **WHEN** 从评估稿生成 V1 manifest
- **THEN** manifest MUST 包含 `下斜史密斯卧推`、`悍马机卧推`、`绳索面拉`、`器械倒蹬机`、`坐姿绳索拉杆二头弯举`、`农夫行走` 和 `壶铃摆荡`

#### Scenario: 清单生成不包含明确移除动作
- **WHEN** 从评估稿生成 V1 manifest
- **THEN** manifest MUST NOT 把 `派克俯卧撑`、`扎特曼弯举`、`JM 推`、`西西深蹲`、`弹力带蚌式`、`TraditionalStrengthTraining` 作为可新建预置动作

#### Scenario: 分类结论可被筛选使用
- **WHEN** 用户按部位或器械筛选动作
- **THEN** `绳索面拉` SHALL 出现在肩部，`罗马尼亚硬拉` SHALL 出现在腿部，`单腿哑铃硬拉` 和 `髋外展` SHALL 出现在臀部，`农夫行走` 和 `壶铃摆荡` SHALL 出现在功能性或其他非解剖入口

### Requirement: 别名兼容与标准名展示

系统 SHALL 提供别名兼容层，用于解析旧 code、旧名称、线上自定义名称和历史计划快照。别名兼容层 MUST NOT 作为第二个用户可见动作库；命中别名时，UI MUST 展示标准动作名。

最低兼容映射 MUST 包含：

| 旧名称/code | 标准动作 |
|---|---|
| `LAT_PULLDOWN`：宽距下拉 / 高位下拉 | 高位下拉 |
| `CABLE_TRICEP_EXT`：绳索臂屈伸 | 绳索臂屈伸 |
| `MACHINE_REVERSE_FLY`：蝴蝶机反向飞鸟 | 蝴蝶机反向飞鸟 |
| `PEC_DECK_FLY`：蝴蝶机飞鸟 / 器械飞鸟 | 蝴蝶机夹胸 |
| `CABLE_CROSSOVER`：绳索十字夹胸 | 绳索夹胸 |
| `DB_ROTATING_CURL`：哑铃轮换弯举 | 哑铃交替弯举 |
| `FACE_PULL`：面拉 | 绳索面拉 |
| `MACHINE_LEG_PRESS`：器械倒蹬(版本2) | 器械倒蹬机 |
| `DB_OVERHEAD_PRESS`：哑铃推肩 | 哑铃推肩 |
| `BB_SQUAT`：深蹲 | 杠铃深蹲 |
| `ROMANIAN_DL`：杠铃罗马尼亚硬拉 | 罗马尼亚硬拉 |

`MACHINE_ROW` MUST 支持 code + name 复合匹配：

- `MACHINE_ROW` + `器械划船1/坐姿器械划船` → `坐姿器械划船`。
- `MACHINE_ROW` + `器械单臂划船` → `单臂器械划船`。

线上自定义动作兼容 MUST 包含：

- `龙门架单边下拉练胸`、`单臂绳索下拉下压` → `单臂绳索下拉`。
- `站姿哑铃划船` → `哑铃直立划船`。
- `坐姿单手牧师凳弯举` → `单臂牧师凳弯举`。
- `牧师凳 附身飞鸟` → `牧师凳俯身飞鸟`。

#### Scenario: 旧 code 展示标准名
- **WHEN** 历史训练记录引用 `LAT_PULLDOWN` 且名称为「宽距下拉」
- **THEN** UI MUST 展示标准动作名「高位下拉」，并能继续展示器械、部位和历史数据

#### Scenario: 合并动作不重复展示
- **WHEN** 用户搜索「绳索十字夹胸」或历史记录引用 `CABLE_CROSSOVER`
- **THEN** 系统 MUST 命中并展示标准动作「绳索夹胸」，不得同时返回一个独立的「绳索十字夹胸」标准动作

#### Scenario: 同 code 不同名称可分流
- **WHEN** 历史计划项引用 `MACHINE_ROW` 且快照名为「器械单臂划船」
- **THEN** 系统 MUST 解析为「单臂器械划船」；当快照名为「坐姿器械划船」时 MUST 解析为「坐姿器械划船」

#### Scenario: 自定义旧名称可解析
- **WHEN** 历史记录中的自定义动作名称为「牧师凳 附身飞鸟」
- **THEN** 系统 MUST 展示标准动作「牧师凳俯身飞鸟」

### Requirement: 线上历史覆盖门禁

系统 SHALL 在实施新动作库 manifest 前运行线上历史覆盖校验。校验 MUST 覆盖训练历史动作引用和训练计划动作引用，且只读取动作 code、动作名称、引用类型和计数，不得导出用户身份、训练日期、重量、次数或备注。

上线门禁公式：

```text
线上历史动作引用 - 新预置动作库 - 别名兼容表 - 明确移除/历史保留清单 = 空集
线上计划动作引用 - 新预置动作库 - 别名兼容表 - 明确移除/历史保留清单 = 空集
```

#### Scenario: 覆盖校验通过
- **WHEN** 覆盖脚本输出训练历史遗漏集合和计划引用遗漏集合
- **THEN** 两个集合 MUST 都为空，才允许上线新 manifest

#### Scenario: 覆盖校验失败阻断上线
- **WHEN** 任一线上动作引用无法被预置动作库、别名兼容表或历史保留清单覆盖
- **THEN** 实施 MUST 停止上线，并补充标准动作、别名映射或历史保留策略后重新校验

#### Scenario: 校验不导出敏感训练数据
- **WHEN** 运行生产库只读校验 SQL
- **THEN** 输出 MUST 仅包含动作引用、code、名称和聚合计数，不包含用户身份、训练日期、重量、次数或备注

### Requirement: 单一动作库搜索与选择体验

系统 SHALL 在动作选择页展示单一预置动作库。搜索 MUST 同时匹配标准名、别名、legacy code 对应名称和线上自定义兼容名称；搜索结果命中别名时 MUST 展示标准动作名。

- 默认筛选 SHALL 包含部位、器械和搜索。
- 胸、肩 SHALL 允许直接显示用户有感知的细分。
- UI MUST NOT 通过“扩展动作”“扩展库”标签区分动作。
- 已明确移除但有历史保留策略的动作 SHALL 不出现在新建训练动作选择结果中；历史详情仍可展示。

#### Scenario: 搜索别名返回标准动作
- **WHEN** 用户搜索「哑铃轮换弯举」
- **THEN** 动作选择页 MUST 返回「哑铃交替弯举」

#### Scenario: 搜索标准名返回标准动作
- **WHEN** 用户搜索「直杆绳索弯举」
- **THEN** 动作选择页 MUST 返回独立动作「直杆绳索弯举」，不得合并到「绳索弯举」

#### Scenario: 移除动作不再可新建
- **WHEN** 用户搜索「TraditionalStrengthTraining」或「沙发伸展」
- **THEN** 动作选择页 MUST NOT 返回可新建的预置动作

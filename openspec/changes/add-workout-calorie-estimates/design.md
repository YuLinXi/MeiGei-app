## Context

现有训练记录以 `Workout` 聚合树为真相源，已完成训练详情页从原始记录即时重算时长、组数和训练量；分享海报同样在客户端根据本地训练记录生成。HealthKit 当前只写入 `.traditionalStrengthTraining` workout，不读取体重，也不写入 active energy。

kcal 在力量训练中误差天然较大。这个能力应定位为个人复盘与海报表达的参考信息，而不是训练记录、同步、Team 或 HealthKit 的权威数据。

## Goals / Non-Goals

**Goals:**

- 用一个可测试的本地纯函数估算训练 kcal。
- 在已完成训练详情页弱展示单次估算结果。
- 在分享海报中重点展示估算 kcal。
- 在「我的 > 训练偏好」提供本地估算配置。
- 保持后端、同步实体、HealthKit 写入行为不变。

**Non-Goals:**

- 不新增 `Workout`、后端表、DTO 或同步字段。
- 不读取 HealthKit 体重。
- 不写入 HealthKit active energy。
- 不做动作级 MET 表或每动作消耗分摊。
- 不在训练进行中页面、首页 hero、列表行、计划页、动作详情页和 Team 动态展示 kcal。

## Decisions

### D1. kcal 估算使用本地派生模型

采用公式：

```text
kcal = MET × 3.5 × bodyWeightKg / 200 × durationMinutes
```

输入来自训练时长、用户设置的估算体重和粗粒度强度。结果四舍五入为整数，展示文案固定使用「约」。

替代方案：动作级 MET 表。放弃原因是当前内置动作数据仍未完全采集，动作级映射会制造“看似精确”的错误结果，并显著扩大维护面。

### D2. 强度使用轻量规则，不开放复杂配置

默认强度为常规力量训练。若训练包含超级组或训练密度较高，则提升为高强度；若完成组密度较低，则降低为低强度。强度判断只影响估算系数，不进入同步数据。

替代方案：让用户每次训练选择强度。放弃原因是会打断训练记录流程，也容易让估算显得过重。

### D3. 体重与显示开关放在本地训练偏好

新增本地偏好：

- 估算体重 kg：默认空，用户设置后才展示 kcal。
- 是否显示消耗估算：默认开启；关闭后训练详情和海报均不展示。

偏好使用 `UserDefaults`，不上传、不参与账号画像。画像域属于服务端权威，训练同步域使用 LWW；本能力不引入新的跨端真相源，符合 Day-1 “统计能重算就重算、少存冗余”的原则。

替代方案：把体重加入 `UserProfile`。放弃原因是体重属于更敏感且会变化的健康信息，当前需求只需本机估算。

### D4. 展示位置按“个人复盘优先、记录中不干扰”处理

- 已完成训练详情页：三联数下方展示一行 `约 xxx kcal · 常规强度`。
- 分享海报：作为重点指标加入海报指标区。
- 训练偏好：提供配置入口。
- 其它页面不展示，尤其训练进行中页面不实时显示。

替代方案：首页/列表展示周消耗。放弃原因是会把产品重心从力量训练表现转向减脂热量。

### D5. HealthKit 保持只写 Workout

估算 kcal 不写入 active energy。HealthKit 是增强项，本地训练记录是真相来源；粗估值不应进入系统健康数据。

## Risks / Trade-offs

- 估算误差较大 → UI 固定使用「约」，并只作为辅助指标展示。
- 用户未设置体重时无结果 → 隐藏展示位，并在训练偏好提供配置入口。
- 海报指标过多 → 海报指标区优先展示 kcal，但保持固定数量和紧凑版式，避免撑破画布。
- 与未来 HealthKit 读取体重冲突 → 当前偏好封装为本地设置，后续可增加“使用 HealthKit 体重”来源而不改训练数据模型。

## Migration Plan

- 无数据库、后端或同步迁移。
- 新偏好首次读取为空；旧用户默认不显示 kcal，直到设置估算体重。
- 回滚时移除 UI 和估算代码即可，遗留 `UserDefaults` 键无副作用。

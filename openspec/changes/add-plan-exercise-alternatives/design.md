## Context

当前 `PlanItem` 用稳定 `itemId` 表示计划动作位，但只保存一个动作；`PlanWorkoutBuilder` 直接把该动作生成 `WorkoutExercise`，并把 `itemId` 写入 `planItemId`。自适应历史目前只按 `planItemId` 精确索引，自适应回写也会把该训练动作的实绩写回原计划项。

备选动作若直接复用这些路径，会导致不同动作串重量，并把备选实绩覆盖到默认动作。训练中临场更换还要求在个人计划被修改/删除、App 重启、离线或 Team 分享版本直接开始后仍能取得候选，因此不能只在界面内临时持有候选。

计划 items 与训练 units 已分别作为同步聚合根内的 JSON 文档保存，继续使用现有 LWW、软删除和幂等同步约定，不建立备选子项的独立同步信封。

## Goals / Non-Goals

**Goals:**

- 普通单动作计划项保存动作引用快照形式的备选列表。
- 训练中在没有任何完成组时临时切换，并在离线、重启和 Team 直接开始路径中保持可用。
- 默认动作与每个实际备选动作的历史重量互不污染。
- 备选实绩不覆盖默认动作处方，也不被自适应回写追加为新计划项。
- 个人副本、Team 分享版本、Fork 和直接开始完整保留候选与重量隐私。

**Non-Goals:**

- 不支持递减组、超级组成员、部分完成后的拆分更换。
- 不保存备选独立处方，不提供排序、提升默认、自动推荐或自动选择。
- 不新增独立同步实体、数据库迁移、REST 接口、依赖或通用 schema 写保护。

## Decisions

### D1：在现有 `PlanItem` 上增加轻量动作快照数组

新增 optional `alternatives`，元素只含 builtin/custom 引用、名称、部位与器械快照。数组顺序就是展示顺序，以 canonical `historyKey` 去重，不增加 UUID、`orderIndex`、同步字段或处方字段。

选择该方案而不是把主动作和备选统一重构为通用 option tree，是为了保留现有主动作字段的 Codable 与跨版本兼容路径，并把 diff 限制在新增可选字段。

### D2：`planItemId` 继续表示父计划动作位

无论最终选择默认还是备选，`WorkoutExercise.planItemId` 都保持父 `PlanItem.itemId`。实际动作身份继续由 builtin/custom/name 计算出的 `historyKey` 表示。

不把备选另设 UUID 写入 `planItemId`，避免改变来源计划项语义、破坏动作位完成判断，以及在复制/Fork/删除后重加时人为切断同动作历史。

### D3：候选与默认落值快照保存到 `WorkoutUnit`

仅对含备选的普通动作单元，在现有 `unitsJSON` 中保存：

- 默认动作与全部备选的动作快照；
- 来源计划模式；
- 训练创建时默认动作已生成的逐组快照。

默认落值快照用于严格模式切回默认动作，并保证原计划随后被修改或删除时仍可恢复本次训练的默认落值。字段均为 optional，旧训练继续按现有逻辑解码。后端 `units` 已是透明 JSONB，无需新列或 DTO 字段。

### D4：更换只在没有任何完成组时发生，并重建未完成组

可更换条件是该 `WorkoutExercise` 所有组均为 `completed == false`，热身组也不例外。更换时保留 exercise/unit/planItem 标识、顺序和动作级休息设置；清空动作备注，并按目标动作重建全部未完成组。

- 严格模式：切回默认动作使用保存的默认落值；选择备选时保留组数、次数、热身结构并清空所有重量。
- 自适应模式：优先读取同一动作位下该实际动作的最近完成历史；无历史时，默认动作使用默认落值，备选使用相同结构但清空重量。

用户选择新动作后统一展示确认，明确未完成输入和备注会被替换，避免静默数据丢失。

### D5：历史使用“动作位 + 实际动作”组合索引

`PlanHistoryLookup` 增加 `(planItemId, historyKey)` 组合键；常规计划预填继续允许现有全局同动作回退，备选切换只使用组合键，避免从其他动作位或默认动作借用重量。数组扫描版历史查询同样要求 `planItemId` 与 `historyKey` 同时匹配。

计划详情的“下次处方”仍表示默认动作路径；用户在训练中主动更换后，本次落值可以不同。

### D6：备选完成只形成历史，不回写默认处方

自适应合并发现 `planItemId` 命中父项、但实际 `historyKey` 与默认动作不同且属于该项备选时：

- 把父项标记为本次已触达；
- 不更新默认动作的 `suggested*` 或 `setPrescriptions`；
- 不追加新 `PlanItem`；
- 不生成“跳过、已保留”回执。

实际训练记录本身已提供下次备选预填所需历史，不另存冗余备选处方。

### D7：共享链路复用现有 JSON 和递归重量脱敏

个人计划副本和 iOS Team 分享快照的手动 `PlanItem` 重建必须显式复制备选。Team 服务端继续原样保存/Fork JSON，并递归删除任何重量字段；Team checkin 仍只从实际 `WorkoutExercise` 生成，不包含未选择候选。

## Risks / Trade-offs

- [旧客户端编辑新计划会丢失未知备选字段] → 接受现有整文档 LWW 的滚动发布限制，在发版说明和跨版本回归中明确；不为单一字段新建 schema 协商系统。
- [训练中更换增加行为矩阵] → V1 只支持普通单动作、无完成组、统一重建与确认，不处理部分完成拆分。
- [动作别名导致重复候选] → 统一使用 `ExerciseLibrary.canonicalHistoryKey` 校验主动作与备选重复。
- [损坏的非必选候选阻断整个计划] → 只禁用无法解析且无名称快照的候选；默认动作校验保持现状。
- [训练创建后的计划内容变化] → 本次候选、来源模式和默认组快照全部随 `WorkoutUnit` 保存，不回查可变计划或 Team 页面状态。

## Migration Plan

1. 先发布能解码 optional 备选和训练单元快照的新 iOS 客户端；后端无需部署变更。
2. 新客户端创建的计划/训练继续通过现有同步接口上传 JSON 文档。
3. 回滚客户端时，旧版本仍可读取默认动作并忽略未知字段；已知限制是旧版本编辑后可能清除备选。
4. 无数据库回填或数据迁移；删除新字段即可回退行为，既有实际训练记录保持有效。

## Open Questions

无。递减组、超级组、备选独立处方和默认提升将在真实使用需求出现后另建 change。

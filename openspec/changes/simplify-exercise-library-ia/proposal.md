## Why

当前动作库已经膨胀到约 1000+ 条，混入大量导入动作、版本动作、左右侧动作、计时型有氧、课程式动作和冷门变体。对《别练了》这种严肃训练记录工具来说，默认动作库应优先帮助用户快速选到主流健身房动作，并保证历史曲线、PR、计划引用稳定。

当前仍处 TestFlight 阶段，线上真实训练用户极少，可以在正式上架前做一次动作库收敛；但必须用线上历史覆盖校验保护已有用户训练和计划。

## What Changes

- 将动作库策略从“大而全导入库”收敛为「单一预置动作库 + 别名兼容 + 明确移除清单」：
  - 用户侧不再区分核心库/扩展库，不展示“扩展动作”概念。
  - 确认保留的热身、拉伸、活动度、功能性动作并入同一个预置动作库。
  - 计时型有氧动作不进入新预置动作库；历史中已出现者保留原名或兼容展示。
- 简化默认浏览 IA：
  - 默认以 `部位 + 器械 + 搜索` 为主。
  - 完整三级解剖树继续作为内部归位和肌肉高亮真源，不作为所有用户的默认导航负担。
  - 胸、肩等用户有明确感知的局部细分可直接展示，如 `胸：上胸/中下胸`、`肩：前束/中束/后束`。
- 固化预置动作库 V1 评估稿：
  - 以 `docs/exercise-library-preset-v1.md` 为本 change 的动作清单、合并清单、移除清单和线上覆盖依据。
  - 标准名、器械、分类、别名兼容和删除策略以后续 manifest 实现为准。
- 引入别名兼容规则：
  - 旧 code / 旧名称 / 用户自定义历史命中别名时，展示标准动作名。
  - 合并动作不得导致历史训练、PR、计划项无法解析。
- 增加线上历史覆盖门禁：
  - 实施前必须对生产库训练动作引用和计划动作引用做只读校验。
  - 任何线上历史引用必须落入新预置动作库、别名兼容表或明确移除策略，遗漏集合必须为空。
- 正式上架前允许清理未发布或低风险导入动作；正式上架后，已发布给真实用户的 `code` 必须冻结。

## Capabilities

### New Capabilities

- 无。

### Modified Capabilities

- `exercise-library`: 调整动作库默认浏览 IA、预置动作库范围、别名兼容、导入库去噪、线上历史覆盖门禁。

## Impact

- iOS：
  - 需要重建内置动作 manifest，读取标准名、器械、分类、肌肉区、别名兼容和移除策略。
  - 动作选择页需要按单一预置库展示，并支持别名搜索命中标准动作。
  - 历史记录、PR、训练计划引用旧 code 或旧名称时必须正常解析并展示标准名。
- 数据：
  - 需要生成预置动作清单、别名兼容表、移除/历史保留清单。
  - 需要增加线上历史覆盖校验脚本或一次性校验流程。
- 后端：
  - 本 change 不要求新增 API、数据库表、同步实体或 Flyway 迁移。
  - 只需要生产库只读查询用于实施前覆盖校验。
- OpenSpec：
  - 修改 `exercise-library` 规格，明确单一预置库、别名兼容、默认浏览和验收门禁。

## Non-goals

- 不做后端动作库下发、远程配置、A/B 实验或用户级开关。
- 不引入拼音搜索、语义搜索、LLM 推荐动作或动作教学内容。
- 不重做肌肉高亮美术，不新增更细的解剖层级。
- 不实现计时型有氧训练模块；跑步、椭圆机、动感单车等不作为新预置动作。
- 不保证继续保留 1000+ 导入动作的全量可见性。
- 不在本 change 中修改训练同步协议、Team API 或账号体系。

## Why

当前递减组被建模为普通动作内部某一组的 `setType`，导致一个动作可以混合普通组与递减组，也暴露出“普通组改递减组 / 递减组改普通组”的转换入口。这个模型和已上线的超级组定义不一致，用户理解上也不够稳定。

本变更将训练单元收敛为三选一：普通组、超级组、递减组。递减组从“组内类型”提升为与普通组、超级组平级的训练单元，并统一热身、统计、添加入口和不可互转规则。

## What Changes

- **BREAKING** 训练单元类型固定为普通组 / 超级组 / 递减组三选一，创建后不可互相转换。
- **BREAKING** 递减组不再作为普通动作内部某个普通 set 的可切换类型出现，不再支持在组级更多菜单中“改为递减组 / 改回普通组”。
- **BREAKING** 超级组现有“解除超级组”为普通动作”的转换语义下线；需要改变类型时，用户删除后重新添加目标类型。
- 递减组作为独立训练单元保存一个动作和有序内部组 segments，不允许普通组内部再单独定义递减组。
- 普通组、超级组、递减组都可以标记为热身组；热身是训练单元上的正交属性，不参与三选一类型。
- 超级组热身按整组/整轮生效，同一轮内两个动作一起标记热身，不支持只给其中一个成员单独标热身。
- 递减组热身按整个递减组生效，标记后其全部内部 segments 都按热身处理。
- 递减组统计从“父级 1 个逻辑组”改为“内部有效 segments 计为 N 组”；训练量、次数、PR 仍按有效 segments 展开。
- 添加入口统一放到底部：保留主按钮“添加动作”用于普通组，另保留一个结构菜单图标，二级菜单提供“递减组 / 超级组”。
- 普通动作卡内部不再展示“递减组”快速添加入口。
- 计划处方、从计划开始训练、自适应回写、保存为计划、Team 分享计划、Fork、训练详情、海报、Team 打卡摘要均按新的训练单元和递减组 N 组统计口径处理。

## Capabilities

### New Capabilities

- 无

### Modified Capabilities

- `workout-tracking`: 递减组从组内 set 类型提升为训练单元；普通组、超级组、递减组三选一且不可互转；三者都支持热身标记；递减组按内部有效 segments 计 N 组；底部添加入口收敛为普通添加动作 + 结构菜单。

## Non-goals

- 不做普通组、超级组、递减组三种类型之间的转换联动。
- 不兼容已上线但尚无人使用的旧递减组数据形态；允许直接重写本地模型、fixtures 与计划 payload 语义。
- 不校验递减组内部重量方向；递增、递减、混合输入仍只作为用户录入的有序内部组保存。
- 不新增后端统计冗余表或独立同步实体；训练统计继续从原始训练记录重算。
- 不引入新的训练单元类型，例如金字塔组、巨人组或复合组。

## Impact

- iOS SwiftData / domain model：`WorkoutUnitKind`、训练单元、普通动作、超级组、递减组、热身标记、统计派生、计划预填与回写。
- iOS 训练 UI：训练进行中列表、底部添加入口、结构菜单、完成按钮、删除入口、热身标记入口、详情展示。
- iOS 计划 UI：计划项编辑、下次有效处方、严格 / 自适应预填、自适应回写、保存为计划、Fork。
- 后端同步与 DTO：Workout 聚合 push/pull、计划 jsonb、Team checkin summary、Team 分享计划 snapshot。
- 派生展示：训练首页周统计、历史曲线、PR、训练结束摘要、分享海报、Team 今日动态与 checkin 详情。
- 测试：需要更新递减组旧口径测试，新增训练单元三选一、热身、N 组统计、添加入口和不可互转覆盖。

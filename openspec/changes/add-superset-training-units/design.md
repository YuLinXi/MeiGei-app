## Context

当前训练记录模型以 `Workout -> WorkoutExercise -> WorkoutSet` 为核心，`WorkoutSetType` 已支持普通组、热身组和递减组。递减组是单动作内部的组形态；超级组不是组形态，而是与单动作平级的一级训练单元。现有计划模型 `WorkoutPlan.items` 也是单动作 `PlanItem` 数组，无法表达「两个动作、统一轮数、按轮完成、轮后休息」的严格超级组结构。

本 change 只做 iOS 本地模型、计划 jsonb、Workout 聚合同步 payload、Team summary 和海报展示的演进，不新增后端写接口或独立同步域。所有统计仍从原始训练记录重算，继续遵守离线优先、聚合 LWW 和软删除约定。

## Goals / Non-Goals

**Goals:**

- 引入显式一级训练单元模型：单动作训练单元与超级组训练单元平级。
- 超级组 V1 固定为 2 个普通正式动作、统一轮数、选择顺序即执行顺序。
- 支持计划详情与训练中添加超级组。
- 支持超级组按轮展示、按轮完成、取消完成、加/减末尾轮、解除超级组。
- 创建/编辑超级组时，统一组数、成员重量和成员次数复用训练录入的自研数字键盘规则。
- 支持超级组训练单元级「轮后休息」，完成某轮后启动休息。
- 统计、PR、历史曲线、Team summary 和海报从超级组成员普通正式组展开计算。
- 计划自适应回写更新超级组成员重量/次数，不自动改变超级组结构或轮数。

**Non-Goals:**

- 不做 3 个及以上动作、循环训练、AMRAP、EMOM、Tabata。
- 不允许超级组内热身组。
- 不允许超级组内递减组，未来也不计划加入。
- 不做超级组嵌套。
- 不做自定义超级组名称 UI。
- 不自动把训练中临时添加的超级组结构写回来源计划。
- 不新增独立后端表、独立同步实体或独立统计实体。

## Decisions

### D1. 使用显式训练单元，而不是 `WorkoutExercise.groupId`

将产品模型定义为：

```text
Workout
└─ units
   ├─ singleExercise
   └─ superset
```

实现时新增 Codable/SwiftData 兼容的训练单元字段，旧 `Workout.exercises` 继续作为单动作兼容入口。新代码的渲染、统计和同步优先通过训练单元视图读取；旧训练没有 units 时按每个 `WorkoutExercise` 派生单动作训练单元。

选择显式训练单元的原因：

- 超级组与单动作平级，不应表现为多个普通动作之间的弱关联。
- Team summary、海报、详情页需要保留结构，弱关联容易在完成后丢失语义。
- 后续如果增加其他一级训练单元，显式模型更自然。

备选方案是只给 `WorkoutExercise` 增加 `groupId/groupOrder`。该方案改动小，但会让 UI、计划、同步和详情持续围绕“动作列表 + 关联标签”补丁化，不符合产品定义。

### D2. 超级组内部成员仍使用动作引用和普通正式组数据

超级组成员包含动作快照、执行顺序、默认重量/次数和每轮 set 数据。每轮完成时，系统同时将两个成员在该轮的普通正式 set 标记为 completed。超级组成员 set 不允许 `.warmup` 或 `.drop`。

统计层不新增超级组统计口径：

```text
完成 4 轮超级组 = 8 个 completed working set
```

训练量、总次数、PR、历史曲线、计划自适应回写均按成员动作和成员 set 展开计算。

### D3. 计划侧升级为 `PlanUnit`

计划 jsonb 从单动作 `PlanItem` 数组演进为一级计划单元数组。为兼容旧数据：

- 解码时优先尝试 `[PlanUnit]`。
- 失败或旧 payload 时继续按 `[PlanItem]` 解码，并包装为 `.singleExercise`。
- 编码新计划时写入 `[PlanUnit]`。

`PlanSupersetUnit` 固定包含两个成员、统一 `roundCount` 和可选 `restAfterRoundSeconds`。从计划开始训练时生成对应 `WorkoutSupersetUnit`，每个成员生成 `roundCount` 个普通正式 set。

计划详情创建/编辑超级组时，统一组数、两个成员的默认重量/次数使用表单版自研数字键盘。重量允许小数，次数与统一组数只允许整数；表单先呈现两个成员动作的重量/次数，最后呈现统一组数。右侧导航用于在「动作 1 重量 → 动作 1 次数 → 动作 2 重量 → 动作 2 次数 → 统一组数」之间切换，末项完成后收起键盘。表单版键盘不得暴露训练中才成立的「加一组」操作。

### D4. 轮后休息属于超级组训练单元

超级组不在创建流程里强制设置休息。它像单动作卡一样，通过右上更多操作配置休息；文案使用「轮后休息」。完成某轮时，如果配置了轮后休息，则启动休息。

落库上 V1 可将计划休息存在超级组单元，实际休息耗时可记录到该轮第二个成员 set，避免新增独立休息实体。UI 和 Team summary 表达为「第 N 轮后休息」。

### D5. 自适应回写只更新超级组成员重量/次数

如果来源计划中已有超级组单元，完成训练后可按成员动作的 completed set 顶组更新该成员默认重量/次数。系统 MUST NOT 自动改变：

- 超级组成员数量或动作选择。
- 成员执行顺序。
- 超级组轮数。
- 单动作与超级组之间的结构关系。

训练中临时添加的超级组只属于本次训练，默认不写回来源计划结构；后续可在完成页提供显式保存入口，但不属于 V1。

### D6. 兼容与同步

Workout 聚合 push/pull 和 Plan jsonb 继续作为既有同步路径，不新增同步域。旧客户端不理解超级组时可能无法完整展示新结构；V1 以当前客户端为兼容边界。服务端如暂不理解 units 字段，需要在 Workout DTO / plan items JSON 中原样透传新增字段。

Team checkin summary 是结构化快照，需要增加训练单元数组；旧 summary 缺少 units 时按旧动作列表展示。

## Risks / Trade-offs

- [Risk] 显式训练单元会触碰模型、同步、计划、统计、详情多处代码，改动面大。
  → Mitigation：保留旧 `WorkoutExercise` / `PlanItem` 兼容解码路径，先让新 UI 通过派生训练单元读写，逐步收敛到单一入口。

- [Risk] 旧计划 payload 与新 `PlanUnit` payload 并存，解码复杂。
  → Mitigation：集中封装计划单元编解码和旧 `PlanItem` 包装，避免视图层直接判断 payload 版本。

- [Risk] 按轮完成会让用户误以为只完成 1 组。
  → Mitigation：超级组卡明确显示「N 轮 · 每轮 2 组」，统计摘要继续展示真实完成组数。

- [Risk] 超级组不支持热身组/递减组可能限制少数高级用法。
  → Mitigation：这是产品边界而非技术缺口；热身和递减组保留在单动作训练单元中。

- [Risk] Team summary 或海报若未更新，会丢失超级组结构。
  → Mitigation：将 summary/poster 展示列入同一 change 范围，完成后保留「超级组 · N 轮」语义。

## Migration Plan

1. 增加 iOS 模型与 Codable 结构，保留旧训练和旧计划兼容读取。
2. 更新计划编辑和开始训练生成逻辑，支持 `PlanSupersetUnit`。
3. 更新训练录入 UI，新增超级组卡片和轮完成。
4. 更新统计、Team summary、海报和详情展示的训练单元展开逻辑。
5. 运行 iOS build 和相关模型测试；旧数据路径应保持可读。

Rollback 策略：如果实现风险过大，可保留 proposal/spec，先只落计划侧结构和模型测试，不发布 UI 入口；未创建超级组的新版本对旧数据无破坏。

## Open Questions

- 超级组完成页是否在 V1 提供「保存本次超级组到计划」入口；当前设计为非目标。
- 超级组可选标题字段是否需要立即入模型但不暴露 UI；当前倾向预留可选字段。

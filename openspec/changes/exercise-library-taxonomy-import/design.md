## Context

别练了内置动作（`BuiltinExercise`，随包只读）现 153 条，字段 `code / name / primaryMuscle(粗8) / equipmentType(6) / primaryRegions / secondaryRegions / formCues`（后三者由 `exercise-muscle-data` 引入）。动作库 UI（`ExerciseLibraryView`）为**单轴**：顶部 7 个粗肌群 chip，无器械筛选、无子分类。

数据源：训记公开标准动作中文名表 `https://github.com/Foveluy/Xunji-movements`（1092 条，README 表格，仅中文名、无内部 key）。经互斥分桶 + 归一化去重估算：拉伸/放松 ~146、有氧/Tabata ~47、弹力带/绳 ~96、TRX/吊环 ~53、颈部 ~14、壶铃 ~14、常规力量 ~600。实际归一化去重跳过 **164** 条，**净新增 928**，库 → **1081**（导入实测值；上述分桶为探索期粗估）。

约束：1–2 人小团队 MVP；iOS 17.4+ SwiftUI；纸感极简设计系统（朱砂红 `Theme.Color.accent`）；内置动作不入同步域；无第三方重依赖偏好。

七项已决策（探索阶段与用户逐条确认）：①全收（含拉伸/有氧/泡沫轴/Tabata）②加器械类型再选收 ③较全量补力量动作 ④浏览轴细分、两套肌群合一 ⑤同名历史自动合并 ⑥code 用语义英文 slug ⑦器械粒度拆分。

## Goals / Non-Goals

**Goals:**
- 动作库从单轴粗筛选升级为「细分部位（含子分类三层）× 器械」双轴，支撑 1000+ 动作的可浏览性。
- 消除「粗 8 浏览 / 细 16 高亮」两套肌群的割裂，统一为一套两级肌群。
- 全量导入训记尚未支持的动作，每条有稳定可读、永久冻结的 `code`。
- 导入不破坏既有用户历史曲线（同名合并、复用旧 code）。

**Non-Goals:**（见 proposal Non-goals）写实配图、饮食、子分类层、置顶、后端下发、即时全量回填 region/要点。

## Decisions

### D1：浏览轴用两级肌群，移除粗 8 类 `MuscleGroup`
新增 `ExerciseCategory` 浏览父级枚举（~15 类），取代 `MuscleGroup`（粗 8）。力量类父级与现有 `MuscleRegion`（16 区高亮叶级）建立**父→子归属**，浏览与高亮自此是同一套肌群的两级；非肌群父级（有氧/热身拉伸/功能性/颈部）无 region 子级。

```
ExerciseCategory(浏览父级 ~15)        MuscleRegion(高亮叶级 16)
├─ 胸    ─────────────────────────▶  chest
├─ 背    ─────────────────────────▶  lats, lowerBack
├─ 肩    ─────────────────────────▶  deltFront, deltRear
├─ 斜方肌 ────────────────────────▶  traps
├─ 二头  ─────────────────────────▶  biceps
├─ 三头  ─────────────────────────▶  triceps
├─ 前臂  ─────────────────────────▶  forearms
├─ 腿    ─────────────────────────▶  quads, hams
├─ 小腿  ─────────────────────────▶  calves
├─ 臀    ─────────────────────────▶  glutes
├─ 腹(核心) ──────────────────────▶  abs, obliques
├─ 颈部  ──┐
├─ 功能性  ├─ 无 region 子级 → 不显高亮
├─ 有氧    │
└─ 热身拉伸┘
```

- 父→子映射用于：未来「按父级聚合高亮」「父级筛选展开子级」等；本 change 主要消费是**浏览筛选**与 **historyKey 归并**职责从 `MuscleGroup` 平移到 `ExerciseCategory`。
- 高亮渲染仍由**每条动作显式的 `primaryRegions`/`secondaryRegions`** 驱动（粒度比父级细），父→子映射不替代它。
- 备选 A：保留粗 8 + 仅加器械轴（被否：粗 8 撑不住 1000+ 浏览，且两套肌群继续割裂）。
- 备选 B：浏览轴直接用 16 区 `MuscleRegion`（被否：`deltFront`/`deltRear`/`obliques`/`lowerBack` 不是自然浏览类目，且无法承载有氧/热身等非肌群品类）。

### D2：器械类型扩展并拆细（6 → ~11）
`EquipmentType` 新增 `史密斯` / `悍马机` / `T杠` / `弹力带` / `悬挂`(TRX/吊环) / `其他`(泡沫轴/徒手拉伸/无器械有氧归此或细分有氧器械)。悍马机/史密斯/T杠 从原「器械/杠铃」拆出。
- 代价：现有 153 条按新粒度**重新打 tag**（如 `SMITH_BENCH_PRESS` 的 `equipmentType` 由「器械」改「史密斯」）。这是用户可见标签变化，但 `code` 不变 → 历史不受影响。
- 「其他」兜底：泡沫轴、徒手拉伸、跑步等无明确器械的归此（或后续细分「有氧器械」；本 change 先用「其他」+「有氧」品类组合表达）。

### D3：code = 语义英文 slug，发布即冻结
格式 `[器械前缀_]动作主体[_修饰]`，全大写蛇形，纯由中文名规则化生成，**与训记序号无关**。
- 器械前缀：杠铃`BB` 哑铃`DB` 史密斯`SMITH` 器械`MACHINE` 绳索`CABLE` 壶铃`KB` 悍马机`HAMMER` 弹力带`BAND` 悬挂`SUSP` T杠`TBAR` 自重(省略)。
- 非力量：拉伸`STRETCH_*` 泡沫轴`FOAM_*` 有氧`CARDIO_*`。
- 唯一性：生成后跑碰撞检测，真重名按训记序号顺序追加 `_2`/`_3`（确定性可复现）。
- 权威源：产出 `中文名 → code` 映射表提交进仓库；code **一经发布永不重算**（它是 historyKey 第一主键，改则用户曲线断裂）。
- 撞现有 153 语义 code 的 → 判为概念重叠，**复用旧 code、不新建**（见 D4）。
- 备选 A：`XJ_<训记序号>`（被否：不可读 + 绑死训记外部编号，序号变/动作下架则语义漂移）。
- 备选 B：`EX_<规范名hash>`（被否：不可读，与现有语义 code 风格割裂；保留为「若将来弃可读性换零人工」的退路）。

### D4：去重 = 复用旧 code；不新建重复条目
训记动作经归一化（去括号注解、去版本/变式后缀、去「机/式/坐姿/单手/宽窄距/负重」等修饰词）匹配到现有 153 条的，判为重叠：保留别练了现有条目（含已回填 region/要点与现名），训记同义变体**跳过不导**。归一化为启发式，产出可疑重叠清单**人工抽查**，不逐条问。

### D5：同名历史自动合并（不可逆，须谨慎）
`historyKey = builtinCode ?? customId ?? name`。导入内置动作名若撞上用户已有**手填/自定义动作名**（其历史按 `name` 归并），执行一次性**本地迁移**：把这些旧记录改挂到新内置动作的 `code`，使历史曲线连续。
- 仅本地迁移，**不新增同步实体**、不改 LWW/幂等。
- 迁移幂等：以「记录的 historyKey 当前等于某新内置动作的 name」为匹配条件，迁移后记录指向 code；重复执行无副作用。
- 风险兜底：迁移前可留旧值快照/日志，便于回滚核对（实现细节，spec 不约束具体存储）。

### D6：数据模型与 Day-1 铁律
- `BuiltinExercise`：随包**只读静态数据**，无 serverId/localId/version——分类字段语义调整（`primaryMuscle`→`category`）与数据扩充均为静态结构变更，不涉及同步/幂等/软删。
- 不新增后端表/接口/Flyway 迁移（内置动作随包发布）。
- 同名历史合并是**本地一次性迁移**，对同步域无新增字段或实体；既有 `Workout`/`CustomExercise` 同步契约不动。

### D7：浏览部位轴支持子分类（三层结构）
部分 `ExerciseCategory` 父级下挂可选**子分类**层，浏览结构为「部位（父级）→ 子分类 → 动作」，对齐训记左栏二级（如「胸」→ 上胸/中下胸）。

**子分类清单（经 1092 条训记动作覆盖校验定稿；计数为粗筛估算，导入精修）：**

| 父级 | 子分类 | 覆盖(约) | 划分依据 |
|---|---|---|---|
| 胸 | 上胸 / 中下胸 | 16 / 120 | 上斜=上胸，平+下斜=中下胸（截图锚点）|
| 背 | 背阔 / 中背 / 下背 | 43 / 79 / 42 | 下拉引体=背阔，划船=中背，硬拉挺身早安=下背 |
| 肩 | 前束 / 中束 / 后束 | 33 / 15 / 9 | 三角肌三束（推举前平举 / 侧平举 / 俯身后束面拉）|
| 腿 | 股四头 / 腘绳肌 | 见消歧 | 深蹲腿屈伸箭步=股四头，腿弯举罗马尼亚直腿硬拉=腘绳肌 |
| 臀 | 臀大肌 / 臀中肌 | ~35 / ~14 | 臀冲桥硬拉=臀大肌，外展后踢蚌式=臀中肌 |
| 核心 | 上腹 / 下腹 / 腹斜肌 / 核心稳定 | 39 / 23 / 26 / 10 | 卷腹 / 举腿提膝 / 转体侧屈 / 平板死虫鸟狗 |
| 有氧 | 稳态有氧 / 间歇·Tabata | 27 / 24 | 跑步单车划船椭圆 vs Tabata波比战绳 |
| 热身拉伸 | 动态热身 / 静态拉伸 / 泡沫轴放松 | 16 / 91 / 39 | 绕环开合 / 拉伸伸展 / 泡沫轴松解滚动 |

**不设子分类的父级**（同质或量小，按肌肉再分无浏览价值；手臂类的差异由器械轴表达）：二头(~54 弯举变体)、三头(~55 臂屈伸/下压变体)、前臂(~16 腕弯举)、小腿(~20 提踵)、斜方肌(~12 耸肩)、颈部(~14)、功能性（异质兜底，导入后视成员再定）。

每个设子分类的父级，最后一个子类为「其余全归此」兜底（如胸的「中下胸」、肩的「前束」），覆盖校验确认 8 个父级**孤儿数均为 0**。

- 模型：`BuiltinExercise` 加 `subcategory: String?`（可空，空=该父级下未细分/「全部」）；`ExerciseCategory` 定义「各父级 → 允许的子分类列表」（无子分类的父级返回空）。
- 子分类**仅父级作用域内有效**（"上胸"只在"胸"下成立），不是全局枚举；筛选时先父级、后子分类层层收窄。
- 子分类与 `MuscleRegion`（高亮叶级）**正交**：子分类服务**浏览收窄**，region 服务**高亮染色**，互不替代（上胸/中下胸的视觉区别仍由 region + 动作名表达）。
- **上表为已审查定稿**：胸→上胸/中下胸以截图为锚点，其余父级经 1092 条覆盖校验（无孤儿）。导入时若发现个别动作归属边界模糊，可单条微调，但子分类集合不再大改。
- **导入消歧规则（覆盖校验发现的真实歧义点）**：
  - 「腿弯举 / 腘绳 / 罗马尼亚 / 直腿硬拉」MUST 先归 **腿 → 腘绳肌**，**优先级高于**「弯举」归二头——否则 `腿弯举` 因含「弯举」被误判进二头（校验中曾致腘绳肌 0 条）。
  - 「耸肩」归 **斜方肌**（独立父级），不归肩；「提踵/小腿」归 **小腿**，不归腿。
  - 关键词匹配按「更具体的部位/动作模式优先」排序，兜底子类放最后。
- 备选：子分类做成独立 `ExerciseCategory` 顶级类目（被否：会让父级轴膨胀到 30+、丢掉"先大类后细分"的层次感，且训记本身就是二级嵌套）。
- 备选：每动作多子分类（被否：浏览收窄需单一归属，多归属让筛选语义含糊；一条动作归一个子分类即可）。

## 依赖与排序

本 change **必须排在** `exercise-muscle-data` 与 `muscle-map-detailed-art` 之后（三者同改 `BuiltinExercise.swift`，并发会交错冲突）：
1. 待 `exercise-muscle-data` 归档 → `exercise-muscle-map` 能力进入 `openspec/specs/`，本 change 才能对其「内置动作细分肌群与要点」Requirement 做 MODIFIED（把 `primaryMuscle` 粗 8 表述改为 `ExerciseCategory`）。
2. 待 `muscle-map-detailed-art` 完成 → 高亮图资产/渲染稳定，导入的力量动作回填 region 后可直接复用。

## 迁移分层（两层交付，不等回填完才上线）

**第一层（本 change 主交付，立即可用）：**
1. 加 `ExerciseCategory` 枚举（含各父级子分类定义）+ `EquipmentType` 扩展；`BuiltinExercise.primaryMuscle`→`category`、加 `subcategory: String?`，编译通过。
2. 现有 153 条迁移到新 `category` + 标 `subcategory`（如卧推系列标上胸/中下胸）+ 重新打 equipment tag。
3. 生成 `中文名→code` 映射 + 导入 928 条（code/name/category/subcategory/equipmentType；非力量 region 留空、子分类多留空）。
4. 动作库 UI 升级双轴 + 子分类展开（部位左栏可展开子分类 + 器械 chip）。
5. 同名历史合并一次性迁移。

**第二层（长尾数据债，分肌群逐批，不阻塞上线）：**
6. 力量类新动作分批回填 `primaryRegions`/`secondaryRegions` + `formCues`（沿用 `exercise-muscle-data` 的回填规范与自写要点规避版权）。回填前这些动作高亮图按既有「空 region → 隐藏」降级，不影响浏览/筛选/记录/计划引用。

**回滚**：分类字段调整为结构性，导入数据为纯静态追加，移除新条目 + 还原 `category`→`primaryMuscle` 即恢复；同名历史迁移留快照可逆。

## Risks / Trade-offs

- [一次导入 928 条数据，质量参差/分类误判] → 按品类分文件 + 人工抽查；导入即可用（名字/筛选），region/要点二层补，错分类可后续单条修订（code 不变则历史安全）。
- [现有 153 条重新打 equipment tag 改变用户可见标签] → 可接受；code 不变、历史不受影响；标签更准反而正向。
- [同名历史合并迁移误并/漏并] → 匹配条件严格（historyKey 当前等于新内置 name 才并）、保留快照、迁移幂等可重跑。
- [移除 `MuscleGroup` 影响现有筛选/引用] → 全量迁移引用到 `ExerciseCategory` + 编译器兜底；筛选行为以「父级筛选」语义对齐旧「粗类筛选」。
- [非力量动作（拉伸/有氧）混入「严肃力量工具」稀释定位] → 用户明确选择全收；以独立浏览品类（有氧/热身拉伸/功能性）隔离，不污染力量类筛选；无高亮、无要点压力。
- [双轴 UI 在纸感极简体系下的密度] → 采用别练了自绘纸感（左栏部位 + 顶部 chip），不照搬训记写实密集卡片。

## Open Questions

- 「其他」器械是否进一步细分「有氧器械」「泡沫轴」？倾向：本 change 先用「其他」+「有氧」品类组合，二期视数据量再拆。
- `ExerciseCategory` 最终类目数与命名（「腹/核心」合一还是分开、「热身」与「拉伸」合一还是分开）需在实现 1.x 时定稿；倾向跟训记左栏对齐但合并到 ~15 以内。
- 各父级的**子分类清单**（D7 表为起点）以训记左栏实际二级定稿；哪些父级设子分类、哪些留空，1.x 时确认。导入 928 条时每条标 `subcategory` 的工作量计入第一层交付（见迁移分层）。
- 同名历史合并是否需用户可感知提示（「已将你过去的 X 记录并入内置动作」）？倾向静默合并 + 可在历史详情看到归并，不打断。

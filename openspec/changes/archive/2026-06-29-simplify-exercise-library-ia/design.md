## Context

当前 `exercise-library` 已经引入 `L1 部位 → L2 肌肉 → L3 肌头/肌区` 的内部分类树，并要求浏览与肌肉高亮同源。但后续动作库规模膨胀到约 1000+ 条，包含大量导入动作、版本动作、左右侧动作、计时型有氧、课程式动作和冷门变体。

本 change 的产品决策已经从“核心库/扩展库分层”调整为“单一预置动作库”。也就是说，用户侧不展示扩展动作概念；确认保留的热身、拉伸、活动度、功能性动作也并入同一个预置库。完整评估稿见 `docs/exercise-library-preset-v1.md`。

关键约束：

- `code` / `historyKey` / 训练计划引用必须稳定；已发布给真实用户的 code 正式上架后不允许改名或删除。
- 当前仍处 TestFlight，正式上架前可以对未发布或低风险导入动作做较彻底收敛，但必须保护线上已有训练历史和计划。
- SwiftData 与同步模型不应因本 change 新增用户数据实体；内置动作仍随 iOS 包发布。
- 后端不参与动作库下发，不新增 API、数据库表、同步实体或 Flyway 迁移。
- 任何线上覆盖校验只能读取动作引用、code、名称、计数，不导出用户身份、训练日期、重量、次数或备注。

## Goals / Non-Goals

**Goals:**

- 用单一预置动作库替代 1000+ 大而全导入库，降低动作选择噪声。
- 保留内部三级解剖树作为归位、高亮和统计依据，但默认浏览不强制暴露完整三级树。
- 以 `docs/exercise-library-preset-v1.md` 固化 V1 标准名、器械、分类、合并、移除和线上兼容结论。
- 为旧 code、旧名称和线上自定义动作建立别名兼容层，命中时展示标准动作名。
- 建立实施前零遗漏校验，确保线上训练历史和计划引用都能被新库解析。

**Non-Goals:**

- 不做后端动作库服务、远程配置、A/B 实验或用户级开关同步。
- 不新增拼音搜索、语义搜索、LLM 推荐、动作教学或动作动图。
- 不重做肌肉图美术，不增加更细解剖层级。
- 不实现跑步、椭圆机、动感单车等计时型有氧记录能力。
- 不把“所有 TestFlight 曾出现过的导入 code”永久冻结为正式动作库承诺。

## Decisions

### D1：用户侧只有单一预置动作库

用户看到的是一个动作库，不再区分“核心动作”“扩展动作”。保留动作、热身动作、功能性动作都进入同一个可搜索的预置库；被移除的计时型有氧或噪声动作不作为新动作展示。

默认动作选择页仍要避免把所有内部树节点塞给用户：

```text
部位：全部 / 胸 / 背 / 肩 / 手臂 / 腿 / 臀 / 核心
器械：全部 / 杠铃 / 哑铃 / 器械 / 绳索 / 史密斯 / 悍马机 / 自重 / 其他
搜索：标准名 + 别名
```

热身、拉伸、活动度、功能性动作可以通过搜索、器械或必要的“其他/功能性”入口访问，但 UI 文案不得使用“扩展库”“扩展动作”。

备选方案是继续保留核心库/扩展库分层。放弃原因：用户已经明确希望不要留下扩展概念，且本阶段更需要先完成动作库内容收敛。

### D2：完整动作清单以文档为评估源，manifest 为实现源

`docs/exercise-library-preset-v1.md` 是产品评估稿，记录标准动作、合并动作、移除动作、线上历史覆盖和 SQL。开发实现时应生成结构化 manifest，例如：

```text
code
displayName
equipmentType
category
subcategory
primaryRegions
secondaryRegions
aliases
legacyCodes
status: preset | removedHistoricalOnly
```

manifest 才是 iOS 运行时读取的实现源；文档用于人工评审和生成 manifest 的依据。

备选方案是把所有清单硬编码在 Swift 文件中。放弃原因：后续还会继续人工校准，结构化 manifest 更容易校验、比对、生成下拉选项和跑覆盖脚本。

### D3：别名兼容层只解决解析与展示，不制造第二个动作库

别名兼容层包含旧 code、旧名称、历史自定义名称到标准动作的映射。它不是用户可见动作列表，也不是扩展库。

解析顺序：

1. 先按标准 `code` 命中预置动作。
2. 未命中时按 `legacyCodes` 命中标准动作。
3. 仍未命中时按 `aliases` / 旧名称命中标准动作。
4. 仍无法命中但历史记录有快照名称时，保留历史快照展示，并计入覆盖校验遗漏。

别名命中后 UI 展示标准动作名。例如：

- `LAT_PULLDOWN`：宽距下拉 / 高位下拉 → 高位下拉。
- `PEC_DECK_FLY`：蝴蝶机飞鸟 / 器械飞鸟 → 蝴蝶机夹胸。
- `CABLE_CROSSOVER`：绳索十字夹胸 → 绳索夹胸。
- `DB_ROTATING_CURL`：哑铃轮换弯举 → 哑铃交替弯举。
- `FACE_PULL`：面拉 → 绳索面拉。

对同一旧 code 可能对应不同标准动作的情况，必须用旧名称分流。例如 `MACHINE_ROW` 下的“器械划船1/坐姿器械划船”映射到坐姿器械划船，“器械单臂划船”映射到单臂器械划船。

### D4：计时型有氧不进入新预置库，历史保留

跑步、椭圆机、划船机、动感单车、跳绳等以计时为主、缺少重量×次数记录价值的动作不进入新预置动作库。历史中已经出现的记录不得丢失：有原始快照则按原名展示；有明确标准映射则按标准名展示。

保留战绳、波比跳、壶铃摆荡、土耳其起立、农夫行走、肩部热身、弹力带绕肩、俯卧臂画圈、猫牛式、下犬式、胸椎旋转、泡沫轴放松等非纯计时但有训练记录价值的动作。

### D5：线上覆盖校验是实施门禁

实施前必须用生产库只读查询得到两类集合：

- 训练历史动作引用集合。
- 训练计划动作引用集合。

两类集合都必须满足：

```text
线上引用 - 新预置动作库 - 别名兼容表 - 明确移除/历史保留清单 = 空集
```

若不为空，不得上线该动作库 manifest。需要先补标准动作、补别名映射，或明确历史保留策略。

### D6：正式上架前和上架后的 code 策略不同

正式上架前：

- 可以重命名、合并、隐藏或删除低风险导入动作。
- 必须保护线上已有训练历史和计划。
- 不要求把所有 TestFlight 出现过的导入 code 都永久冻结。

正式上架后：

- code 一经发布给真实用户，即冻结。
- 后续只能新增动作、补别名、调整显示名或分类，不能删除或复用旧 code。

## Risks / Trade-offs

- [Risk] 删除或合并动作导致历史无法解析 → 用线上覆盖校验做上线门禁，遗漏集合必须为空。
- [Risk] 别名合并后 PR/历史曲线分组不符合预期 → 实现时明确标准动作解析结果，并增加代表性旧 code 的回归测试。
- [Risk] 单一预置库仍然过大 → 默认浏览控制在部位、器械和搜索，不暴露完整三级树；后续可继续从清单中移除低价值动作。
- [Risk] `MACHINE_ROW` 这类旧 code 需要按名称分流 → 兼容表必须支持 code + name 的复合匹配，不得只按 code 单字段粗暴映射。
- [Risk] 正式上架前清理边界不清 → tasks 中必须增加“已发布/线上引用覆盖”校验，校验不过不得删除。

## Migration Plan

1. 以 `docs/exercise-library-preset-v1.md` 生成结构化预置动作 manifest、别名兼容 manifest 和移除清单。
2. 在 iOS 动作库加载层接入新 manifest，提供标准动作解析、别名解析和历史快照降级展示。
3. 调整动作选择页默认浏览为单一预置库，不显示核心/扩展概念。
4. 跑生产库只读覆盖校验；遗漏为 0 后再进入真机/模拟器验证。
5. 若上线后发现选择体验退化，回滚方式是恢复上一版随包动作 manifest；由于不改后端 schema 和同步模型，回滚只涉及 iOS 包内资源与加载逻辑。

## Open Questions

- V1 manifest 的 `code` 命名是否全部沿用现有 code，还是对少量未发布导入动作生成新 code，需要实施时结合当前 iOS 代码和线上覆盖结果决定。
- 对历史中已出现但被明确移除的计时型有氧动作，是否只保留历史展示，还是允许从历史详情再次添加同名自定义动作，需要 UI 实现时确认。

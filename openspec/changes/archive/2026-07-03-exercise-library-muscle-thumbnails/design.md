## Context

动作库和添加动作抽屉共用 `ExerciseLibraryContentView`。内置动作行当前左侧为 40pt 首字头像，右侧展示动作名、部位、细分和器械；动作详情页另行展示完整 `MuscleMapView`。

内置动作数据已经包含 `primaryRegions` / `secondaryRegions`，并通过 `ExerciseCategory` 的解剖树与浏览归位同源。列表场景需要快速识别训练区域；当当前肌肉图能视觉区分主动肌区域时，缩略图可以比 L1 大类更细，但不伪造 SDK 无法表达的肌头差异。

本 change 仅影响 iOS 本地展示层。数据模型、同步字段、幂等键、软删除和后端接口均不变，因此 Day-1 铁律在本设计中的体现是：不新增同步对象、不新增服务端写接口、不迁移已有本地或服务端数据。

## Goals / Non-Goals

**Goals:**

- 在动作库和添加动作抽屉中，让内置动作行能直接展示所属解剖大类或可区分主动肌区域。
- 复用现有 `MuscleMap` 渲染和 `ExerciseCategory.regions` 大类数据。
- 将列表缩略图离线导出为 PNG asset，避免大量列表行实时绘制 SDK view 或运行时生成 `UIImage`。
- 对缺少肌群数据、非解剖类动作或自定义动作保持自然降级。

**Non-Goals:**

- 不新增每个动作的照片、动图、教学图或独立图片资源。
- 不改变动作详情页的信息架构。
- 不新增自定义动作肌群编辑能力。
- 不调整后端、SwiftData schema、同步协议或历史归并逻辑。

## Decisions

### 1. 新增列表专用紧凑肌群缩略图

新增 `MuscleMapThumbnail`，与 `MuscleMapView` 同文件或相邻文件实现。它只展示一个人体面、隐藏图例、固定用于小尺寸头像位。

理由：

- 详情页 `MuscleMapView` 是正背并排 + 300pt + 图例，直接缩小会不可读。
- 列表用户目标是快速识别“主要练哪个大类”，不是完整阅读动作级主动肌/协同肌说明。

备选方案：

- 直接复用 `MuscleMapView`：实现简单，但列表中正背挤压严重，且每行实时绘制成本高。
- 每个动作独立图片：可读性高，但资产采集和版权成本过高。

### 2. 有限可区分分组图，按可渲染区域复用

缩略图在列表层不按每个动作生成独立图，而是按当前肌肉图可视觉区分的训练分组生成有限图片。默认使用 `ExerciseCategory.regions` 大类图；如果动作的 `primaryRegions` 在当前 SDK 中能形成更清晰的视觉差异，则使用这些主动肌区域作为缩略图 key。不能被当前 SDK 区分的子分类继续使用大类图，例如胸的「上胸 / 中下胸」都回退胸大类；肩后束可通过背面三角肌区分，肩中束暂不伪造成独立图。

理由：

- 用户在列表里需要识别足够明确的训练区域，但 48pt 缩略图不能承诺 SDK 未提供的肌头级精度。
- 有限可区分分组能提升辨识度，同时保持图片生成与缓存数量可控。

### 3. 离线导出 PNG，列表不实时绘制 SDK view

`tools/muscle-thumbnail-exporter` 使用 `MuscleMap` SDK 和与 App 相同的裁切参数，将有限分组导出为 48pt @3x PNG imageset。App 运行时的 `MuscleMapThumbnail` 只接收 asset name 并通过 `Image(assetName)` 展示，不调用 `ImageRenderer`、不维护 `NSCache`、不在列表行中实例化 SDK view。

理由：

- `MuscleMap` SDK 是 SwiftUI/Canvas/Path 渲染，直接放进大量列表行会增加滚动开销。
- 离线导出有限分组图片后，长列表滚动主要是普通 asset bitmap 绘制，首屏也不需要等待首次生成。

### 4. 单面展示，按大类区域自动选正面或背面

缩略图根据大类 region 优先选择高亮区域更多的一侧；正背数量相同时默认正面。`both` 区域两侧都计入。

理由：

- 胸、腿等默认显示正面；背默认显示背面。
- 列表没有空间放切换控件，也不应要求用户交互才能看懂。

### 5. 局部裁切只作为视觉聚焦，不改变数据语义

缩略图可按大类 region 粗分为上半身、躯干、下半身或全身焦点，通过缩放和裁切增强可读性。上半身图采用 bust close-up：头、胸、肩、上臂占据主要画面，胸/肩高亮不贴顶、不缩成全身小人。其它大类也遵守同一原则：高亮区域和相邻解剖参照应填满头像位的大部分空间。裁切规则只影响视觉，不改变肌群映射、浏览归位或详情页展示。

理由：

- 40-52pt 内全身人体对腿、臀、肩等区域不够清晰。
- 局部展示符合“选择动作时不用进详情也能看清主要肌群”的目标。

### 6. 降级策略沿用现有首字头像

当 `primaryRegions` 为空、动作为自定义动作或非解剖动作不适合高亮时，继续显示当前首字头像。

理由：

- 避免空人体占位误导用户。
- 不阻塞热身拉伸、自定义动作和未来未回填动作的浏览与选择。

## Risks / Trade-offs

- [风险] 缩略图参数调整后可能忘记重导出 PNG → [缓解] 保留 `tools/muscle-thumbnail-exporter`，并在任务与文档中记录重生成命令。
- [风险] 当前 SDK 无法区分部分用户熟悉的细分区，例如上胸/中下胸和肩中束 → [缓解] 不伪造不支持的子分类；保留大类图，并通过列表文字与详情页表达细分名称。
- [风险] 细分肌头在 SDK 中存在回退，例如背阔、肩中束、臀中肌 → [缓解] 仅对可形成明显视觉差异的区域使用细分缩略图，其余继续用大类图。
- [风险] 裁切规则可能对少数复合大类不完美 → [缓解] 以解剖大类为准选择主焦点，详情页仍提供完整正背图。

## Migration Plan

1. 添加紧凑缩略图组件和有限分组 asset 映射。
2. 离线导出男/女两套 PNG imageset。
3. 自定义动作行保持现状。
4. 编译验证 iOS target。

回滚方式：恢复内置动作行左侧为原首字头像；无数据迁移和服务端回滚。
